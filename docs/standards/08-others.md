# 补充开发规范（通用，语言无关）

覆盖安全、测试、配置、CI/CD、性能、依赖管理、代码评审等跨领域强约束。

## 1. 安全（MUST）

- 密钥 / Token / 证书 **MUST** 走密钥管理（Vault / KMS / 环境变量注入），**禁止**入库或硬编码。
- 外部输入 **MUST** 校验与转义，防范注入（SQL / NoSQL / 命令 / XSS / LDAP）。
- 使用参数化查询 / 预编译语句；禁止字符串拼接 SQL。
- 依赖 **MUST** 定期扫描漏洞（SCA），高危漏洞修复前不得发版。
- 最小权限原则：服务账号 / IAM 仅授予必要权限。
- 敏感日志 **MUST** 脱敏（见日志规范）；禁止打印密码、Token、完整卡号、身份证。

## 2. 测试（MUST）

- 强制 TDD：先写失败测试（RED）→ 最小实现（GREEN）→ 重构。
- 测试分层：单元 / 集成 / 端到端；**MUST** 有稳定可重复的单元测试。
- 覆盖率门槛：**新增代码行覆盖 ≥ 80%**（关键路径 100%）；集成测试覆盖主流程。
- 测试 **MUST** 独立、可并行、无外部依赖副作用（用测试替身 / 容器化依赖）。
- 禁止为刷覆盖率写无断言的"假测试"。
- 测试命名表达意图（`Should_ReturnX_When_Y`）。

## 3. 配置（MUST）

- 遵循 12-Factor：配置与代码分离，通过环境变量 / 配置中心注入。
- 不同环境（dev/test/staging/prod）**MUST** 配置隔离，prod 配置不进仓库明文。
- 配置变更 **MUST** 可回滚；敏感配置加密存储。
- 禁止把环境相关地址 / 开关写死在代码里。

## 4. CI / CD（MUST）

- 每次 push / PR **MUST** 触发 CI：format → lint → 单测 → 构建 → 安全扫描。
- CI 失败 **MUST** 阻断合并（状态门禁）。
- 部署 **SHOULD** 自动化、幂等、可观测；禁止手工 scp 式发布。
- 流水线 **SHOULD** 分层（快反馈的 lint/单测前置，慢的集成/性能后置）。

## 5. 性能与稳定性（SHOULD）

- 外部调用 **MUST** 设超时与重试（带退避），禁止无限等待。
- 资源（连接 / 线程 / 协程）**MUST** 有上限与回收，防泄漏。
- 关键路径 **SHOULD** 有性能基线，回归超阈值告警。
- 批量 / 循环 **SHOULD** 评估复杂度，避免 N+1、全表扫描。

## 6. 依赖管理（MUST）

- 依赖 **MUST** 显式声明并锁版本（lockfile：`go.sum` / `pom.xml` 锁定 / `package-lock`）。
- 禁止引入未审计的未知来源依赖；新增依赖 **SHOULD** 说明理由。
- 定期更新依赖，处理废弃 / 漏洞；大版本升级 **MUST** 评估破坏性。

## 7. 可观测性（SHOULD）

- 关键操作 **SHOULD** 有指标（RED：请求数 / 错误率 / 时延）、日志、分布式追踪（traceId）。
- 错误 **SHOULD** 可聚合、可告警；指标命名一致。

## 8. 代码评审（MUST）

- 所有合并 **MUST** 经至少 1 人评审（或 AI + 人复核）。
- 评审关注：正确性、规范符合度、安全、测试、可读性、边界。
- 评审意见 **MUST** 逐条解决或显式记录；禁止"已解决"但未改。

## 检查清单

- [ ] 无密钥硬编码；外部输入已校验
- [ ] 新增测试覆盖 ≥ 80%，TDD 顺序正确
- [ ] 配置外部化、环境隔离
- [ ] CI 门禁完整（含安全扫描）
- [ ] 外部调用有超时 / 重试；资源有上限
- [ ] 依赖锁版本、无未知来源
- [ ] 评审已完成且意见闭环
