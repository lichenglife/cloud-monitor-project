## 一、P0 级（阻断开工，须会议当场决议）

### D1 前端技术栈选型（框架二选一）

- **背景**：系统设计说明书 §2.7 定"前端自研 Vue3 / React + TypeScript"，二选一未决；DD-15 v0.2 已给双基线，但框架未定则页面/Mock/组件库全无法开工。
- **可选方案**
  - **方案 A — React 18 + TS + Vite + Ant Design + AntV（G6/ECharts）**
  - **方案 B — Vue 3 + TS + Vite + Element Plus/Ant Design Vue + AntV**
  - **方案 C — 多应用分栈**（大屏用 Vue、管理后台用 React 等）
- **差异与影响**

| 维度 | A（React） | B（Vue3） | C（分栈） |
|---|---|---|---|
| 可视化生态 | 最成熟（Grafana/Jaeger UI 均 React；G6/ECharts/d3 集成案例多） | 良好（AntV 原生 Vue 支持） | 各取所长但双栈维护成本翻倍 |
| 团队匹配 | 若团队偏 React 则效率高 | 若团队偏 Vue 则效率高 | 需双栈人力 |
| 重型图（拓扑/火焰） | 现成 hooks 封装、memo 防重渲染成熟 | Composition API 同样胜任 | 同 A/B 但重复建设 |
| 风险 | 团队不熟则踩坑 | 团队不熟则踩坑 | 一致性差、包体/规范分裂 |
| 决策影响 | 决定全部页面/Mock/CI 基线 | 同左 | 需双倍基建 |

- **推荐**：无强团队偏好时**推荐 A（React）**——可观测性领域生态与可视化库集成最成熟；若团队主力为 Vue 则选 B。C 不推荐（治理成本过高）。
- **建议决议人 / 时限**：前端负责人 + 架构负责人 / **会议当场**。
- **关联文档**：DD-15 v0.2 §1、系统设计说明书 §2.7。

### D2 拓扑万级节点渲染方案

- **背景**：DD-16 / DD-15 均标注万级节点为待评审；渲染方案直接关系 P95<1s 与可用性。
- **可选方案**
  - **方案 A — G6 v5 WebGL 渲染 + 视口 LOD + 按 scope 聚类收起 + Web Worker 布局**
  - **方案 B — 默认聚类收起（按中心/机房聚合），仅下钻展开，不做全量万节点渲染**
  - **方案 C — 专用 GPU 渲染库（cosmograph 等）自建布局**
- **差异与影响**

| 维度 | A（G6 WebGL+LOD） | B（聚类优先） | C（GPU 专用库） |
|---|---|---|---|
| 全局拓扑可视 | 强（万节点可呈现） | 弱（默认只见聚合） | 最强（极限性能） |
| 性能(P95) | 达标（WebGL+LOD+Worker） | 最稳（节点少） | 极致但自研 |
| 复杂度/工期 | 中（G6 成熟，需 PoC 2–3d） | 低 | 高（自研布局/交互） |
| 依赖风险 | G6 v5 WebGL 成熟度需验证 | 低 | 生态小、维护难 |
| 决策影响 | 拓扑页核心体验 | 弱化"全局视图"价值 | 长期维护负担 |

- **推荐**：**方案 A**（先 PoC 验证万节点帧率与 Worker 布局耗时）；B 作为降级兜底共存（超阈值自动切聚类）。C 不推荐（投入产出比低）。
- **建议决议人 / 时限**：前端负责人 + 架构 / **会议当场定方向，PoC 2 周内出结论**。
- **关联文档**：DD-15 v0.2 §7、DD-16 §5。

## 二、P1 级（重要，应本阶段决议）

### D3 采集采样率默认值（试点期）

- **背景**：DD-07/DD-14"头部默认 sample_rate 初值"待定；影响成本与排查盲区。
- **可选方案**
  - **方案 A — 试点 100% 采样**（保数据完整）
  - **方案 B — 按比例采样（如 10%~50%）+ 错/慢必采**
  - **方案 C — 按流量分级（低峰 100% / 高峰降采样）**
- **差异与影响**

| 维度 | A（100%） | B（按比例） | C（分级） |
|---|---|---|---|
| 存储成本 | 最高（~105TB/年压缩全量） | 低（降至 1/10~1/2） | 中（高峰降本） |
| 排查盲区 | 无 | 有（抽样链路缺失） | 低峰无、高峰有 |
| 实施复杂度 | 最低 | 低 | 中（需流量感知） |
| 决策影响 | 成本换可见性 | 省成本但有盲区 | 折中 |

- **推荐**：**方案 A 试点期**（数据完整、验证全链路），稳定后转 B/C（呼应 DD-14 成本治理）。
- **建议决议人 / 时限**：架构 + 业务方 / 会议或 1 周内。
- **关联文档**：DD-07 §3、DD-14 §2。

### D4 自适应采样策略与触发机制

- **背景**：DD-14 §2.2 自适应采样进阶能力待定触发与步进。
- **可选方案**
  - **方案 A — 规则触发调档**（错误率↑→采样↑；流量↑→采样↓；步进 ±10%）**
  - **方案 B — 闭环控制**（基于百分位/同环比的 PID 类连续调节）**
  - **方案 C — 暂不启用，仅静态配置**
- **差异与影响**

| 维度 | A（规则调档） | B（闭环 PID） | C（静态） |
|---|---|---|---|
| 可控/可解释 | 强（规则透明） | 弱（黑盒调参） | 最强 |
| 平滑度 | 有台阶跳变 | 平滑 | 无弹性 |
| 复杂度 | 中 | 高（需调参） | 低 |
| 成本弹性 | 好 | 最好 | 无 |

- **推荐**：**方案 A（一期）**，B 作二期演进，C 为兜底。
- **建议决议人 / 时限**：架构 / 1 周内。
- **关联文档**：DD-14 §2.2。

### D5 尾部采样缓冲窗口与在途上限数值

- **背景**：DD-07 v0.2 已量化 ~100–300MB@10s，但 `window`(10s) 与 `max.inflight`(20000) 仍为待评审数值。
- **可选方案**
  - **方案 A — window=10s，in-flight=20000**（已写入推荐值）
  - **方案 B — window=5s，in-flight=10000**（更省内存、错/慢捕获略降）
  - **方案 C — window=30s，in-flight=50000**（捕获最全、内存压力大）
- **差异与影响**

| 维度 | A(10s/20000) | B(5s/10000) | C(30s/50000) |
|---|---|---|---|
| Collector 内存 | ~100–300MB | ~50–150MB | ~300–800MB |
| 错/慢链路捕获率 | 高 | 中（短 Trace 易漏窗） | 最高 |
| 风险 | 可控 | 低 | OOM 风险需更强保护 |

- **推荐**：**方案 A**（已在 DD-07 v0.2，会议确认数值）；长 Trace 多中心场景可后续调到 15s。
- **建议决议人 / 时限**：架构 / 会议或 1 周内。
- **关联文档**：DD-07 §3.2。

### D6 告警评估引擎与异常检测算法

- **背景**：DD-06 v0.2 已定 alert-engine 消费预聚合流 + 滑动窗口，但"异常检测算法选型"仍待定。
- **可选方案**
  - **引擎**：X 流式消费预聚合指标流 / Y 定时轮询 ES
  - **算法**：① 静态阈值 ② 同环比/EWMA ③ 机器学习
- **差异与影响**

| 维度 | 静态阈值(①) | 同环比/EWMA(②) | ML(③) |
|---|---|---|---|
| 检出延迟 | 低（P0≤10s） | 中 | 中 |
| 误告率 | 中（需调阈值） | 低（自适应基线） | 最低 |
| 数据积累 | 无需 | 需历史 | 需大量标注 |
| 复杂度 | 低 | 中 | 高 |
| 推荐期 | 一期 | 二期 | 远期 |

- **推荐**：引擎用 **X（流式）**；算法 **①一期 → ②二期 → ③远期**。
- **建议决议人 / 时限**：架构 + 运维 / 1 周内。
- **关联文档**：DD-06 §7.5、DD-05。

### D7 TraceID 位数与节点位设计

- **背景**：DD-14 §3 草案为 128bit（时间戳41+中心3+节点10+序列10）；节点位是否够 5000+ 工程实例待确认。
- **可选方案**
  - **方案 A — 128bit（中心3 + 节点10bit=1024/中心）**（已推荐）
  - **方案 B — 128bit（节点扩至 12bit=4096/中心）**
  - **方案 C — 64bit（省空间，跨中心拼接弱）**
- **差异与影响**

| 维度 | A(128/10bit) | B(128/12bit) | C(64bit) |
|---|---|---|---|
| 防碰撞 | 强（序列10bit+时间） | 强 | 弱（高位缺失） |
| 跨中心定位 | 强（内嵌中心位） | 强 | 弱 |
| 节点位容量 | 1024/中心（作来源标识足） | 4096/中心 | — |
| 存储/索引开销 | 略增 | 同 A | 最小 |
| 结论 | 节点位=来源标识非实例主键，够用 | 更宽裕 | 不推荐 |

- **推荐**：**方案 A**（节点位仅用于来源标识，实例精确识别靠 `service_instance`/`host_ip`）。
- **建议决议人 / 时限**：架构 / 会议或 1 周内。
- **关联文档**：DD-14 §3。

### D8 多中心 Kafka 跨 AZ 副本策略

- **背景**：DD-08 决议 A 集中上海 DC1，但副本是否跨可用区未定。
- **可选方案**
  - **方案 A — 关键 Topic RF≥3 跨 AZ 副本**（上海 DC1 内跨可用区）
  - **方案 B — 仅同 AZ 副本**
  - **方案 C — 苏州建异步灾备副本（异地）**
- **差异与影响**

| 维度 | A(跨AZ) | B(同AZ) | C(异地灾备) |
|---|---|---|---|
| 单 AZ 故障 | 不丢 | 丢（该 AZ 全挂） | 不丢+异地 |
| 带宽/成本 | 中（区内） | 低 | 高（跨城） |
| 复杂度 | 低 | 最低 | 高 |

- **推荐**：**方案 A**（区内跨 AZ，性价比最优）；C 作二期异地容灾评估。
- **建议决议人 / 时限**：架构 + 运维 / 1 周内。
- **关联文档**：DD-08 §4。

### D9 ES 冷层存储介质选型

- **背景**：DD-04/DD-10"冷层存储介质"待定（1 年留存）。
- **可选方案**
  - **方案 A — 热 SSD + 冷层对象存储(OSS/S3) 快照（ILM 自动转）**
  - **方案 B — 热 SSD + 大容量 HDD 本地冷层**
  - **方案 C — 仅降采样不保留冷明细**
- **差异与影响**

| 维度 | A(对象存储) | B(本地HDD) | C(仅降采样) |
|---|---|---|---|
| 成本 | 最低 | 中 | 最低（但丢明细） |
| 查询速度(冷) | 慢（需回载） | 中 | 无明细可查 |
| 运维 | 低（托管） | 中（自管盘） | 低 |
| 合规留存 | 满足 | 满足 | 不满足（丢原始） |

- **推荐**：**方案 A**（成本最低且满足 1 年留存合规；冷数据查询少，慢可接受）。
- **建议决议人 / 时限**：DBA + 架构 / 1 周内。
- **关联文档**：DD-04 §3、DD-10 §3。

### D10 压测目标倍数

- **背景**：DD-10"压测 10× 流量目标是否过高"待定。
- **可选方案**
  - **方案 A — 试点期 3×，上线前 10× 全量**
  - **方案 B — 全程 10×**
  - **方案 C — 按真实峰值 + 50% 余量**
- **差异与影响**

| 维度 | A(3×→10×) | B(10×) | C(峰值+50%) |
|---|---|---|---|
| 资源消耗 | 中 | 高 | 低 |
| 风险覆盖 | 试点验证降级，上线全量 | 最严 | 仅余量 |
| 推荐 |

- **推荐**：**方案 A**（试点用 3× 验证背压/降级，上线前 10× 全量压测出具报告）。
- **建议决议人 / 时限**：测试 + 架构 / 排期确认。
- **关联文档**：DD-10 §3。
