## 1. 采集架构总览（决议 A 集中上海 DC1）

```
业务应用（Java=SkyWalking Agent / Go=OTel Agent）
   │ OTLP gRPC:4317 / HTTP:4318（biz 直连；dmz 经代理转发）
   ▼
Collector（接收→解析→采样→转换→导出，无状态可 HPA）
   │ 幂等去重(trace_id+span_id) + 背压
   ▼
Kafka（上海 DC1：otel-spans / otel-metrics / otel-logs / otel-infra-metrics）
   │
   ▼
消费侧（topology-calc / 聚合服务 / Flink 二期）→ ES 6 索引 + MySQL + Redis
```

- **多中心**：上海 DC1 / 苏州 DC2 各中心 Collector 本地接收与一级处理，统一汇聚至上海 DC1 Kafka（决议 A）；biz/dmz 网络隔离下 dmz 经代理转发（决议 A）。
- **无状态**：Collector 不持有状态，水平复制（X 轴），K8s HPA 按 CPU/吞吐扩缩。

## 3. 采样双模（核心：采样即正义）

### 3.1 头部采样（Head-based，Agent 侧）
- 在请求入口即时决策，**不持有完整 Trace**，开销最低。
- 决策依据：低基数键（`http.status_code`、`db.operation`，见 DD-03.5）、`sample_rate`（来自 `agent:config`，Redis #1）。
- 默认 `sample_rate` 由 `t_agent_config` 下发（热配置），普通流量按比例（如 10%~100%）；**评审决议 D3=方案A：试点期默认 100% 全量采样**，保数据完整、验证全链路，稳定后按成本下调（呼应 DD-14 成本治理）。

### 3.2 尾部采样（Tail-based，Collector 侧）
- 持有 Trace 完整 Span 后决策，**错误/慢请求必采**。
- **强制保留规则（v1）**：
  - 含错误 Span（`status.code=1` / `error=true`）→ 100% 采样；
  - 慢请求（Trace 总耗时 > **2s**，可配）→ 100% 采样；
  - 超长 Trace（> 50,000 spans，DD-03 §9.4）→ 采集侧硬截断并计数，仍保留。
- **尾部采样缓冲**：Collector 按 `trace_id` 分组缓冲窗口 `tail.sample.window`（默认 **10s**，评审决议 D5=方案A），窗口内收齐同 Trace Span 后决策导出。
- **内存量化（后端架构评审 C5）**：试点峰值 ~1万 span/s、平均 ~20 span/Trace → 约 500 在途 Trace/s × 10s 窗口 ≈ **5000 在途 Trace**，每 Trace ~20 Span × ~1KB ≈ **~100MB** 常驻；叠加对象头/索引开销，峰值约 **100–300MB/Collector 实例**。Collector 堆需据此预留（建议 ≥1GB 堆，尾部缓冲占 ≤30%）。
- **在途上限保护**：配置 `tail.max.inflight.traces`（默认 20000，评审决议 D5=方案A），超出时**丢弃最旧在途 Trace 的缓冲并计数告警**（DD-06），避免内存无界增长致 OOM。

### 3.3 自适应采样（Adaptive，进阶，呼应 DD-14）
- 基于流量/错误率/耗时百分位动态调整 `sample_rate`：错误率升高时自动调高采样率（保错误链路）、流量洪峰时自动调低（控成本）。
- v1 以"静态配置 + 规则触发调档"起步；完整自适应算法见 DD-14。

## 4. Collector 处理流水线

1. **接收（Receive）**：OTLP gRPC/HTTP，TLS + Agent 令牌鉴权（零信任）。
2. **解析（Parse）**：解码 Avro/OTLP，按 DD-03.5 语义校验、补全多中心维度（`data_center/network_area/room/cluster`）。
3. **去重（Dedup）**：`trace_id+span_id` → Redis `trace:dedup:{traceId}`(#4，TTL 10m)，幂等（DD-02）。
4. **采样决策（Sample）**：头部先决 + 尾部决策（错/慢必采）。
5. **转换与脱敏（Transform）**：PII 正则掩码（`db.statement`/`http.url`/自定义 Attribute，呼应 DD-03.5 §3、DD-14 ETL）。
6. **导出（Export）**：Span/Metric/Log/Infra 分 Topic 写入 Kafka（上海 DC1）；ES 不可用时降级缓冲本地磁盘队列，不反向压垮 Agent（可靠性）。

## 5. 背压与降级（可靠性红线）

- Collector 入口限流：试点峰值 QPS 500 × 系数 3 ≈ **1500 QPS** 预留（DD-01 §9 定稿），超阈触发背压。
- 背压传导：Kafka 消费 Lag 升高 → 聚合服务降速；ES 写阻塞 → Collector 本地磁盘缓冲（有界队列，防 OOM）。
- 降级优先级：**保采集 > 保聚合展示**；存储压力大时允许丢弃非错误/非慢链路 Trace（DD-04 成本原则）。

## 6. 热配置下发（≤30s）

- 配置源 `t_agent_config`（DD-02）→ config-center 发布 → Redis `agent:config:{clusterId}`(#1) + `config:version:{clusterId}`(#11)。
- Agent Watch 拉取，≤30s 生效采样率/最大 TPS（`sampling_mode=fixed/adaptive`）；极端场景（CPU 打满）Agent 自动熔断/关闭（DD-13）。

## 7. 数据流转与多中心

- 各中心 Collector 汇聚 → 上海 DC1 Kafka（决议 A）；跨中心 Trace 由 `trace_id` 全局唯一拼接（DD-14 TraceID 策略 + DD-08 多中心）。
- Kafka 分区按 `trace_id` hash 保序（DD-02 §4）；消费端幂等去重。

## 8. 设计原则遵循声明

- 采样即正义✅（双模 + 错/慢必采 + 自适应） 资源消耗红线✅（Agent CPU<1%/内存<128M，DD-13；头部采样低开销） 可靠性✅（背压/降级/幂等去重/本地缓冲） 无状态✅（Collector HPA） 零信任✅（Agent 令牌 + TLS） 控制基数✅（采样降基数、脱敏降基数） 向后兼容✅（Avro 版本化、DD-03.5 兼容） 防御式编程✅（解析校验）。
