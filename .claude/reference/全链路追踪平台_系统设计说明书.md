## 1 概述

### 1.1 编写目的
本文档为**全链路追踪分析平台一期**的详细系统设计说明书，旨在：
- 明确系统的总体架构、部署拓扑、服务拆分与技术选型，作为研发实施的设计依据；
- 对接评审决策（议题 A–E），将部署架构、存储选型、采集方案等决策转化为可落地的设计方案；
- 为各功能模块提供功能说明、流程逻辑实现、数据结构（Redis/DB/MQ）、页面交互与安全设计的详细规格；
- 作为后续开发、测试、运维的统一参考基准。

### 1.2 适用范围
- **适用对象**：技术负责人、后端研发、前端研发、SRE/运维、测试、安全合规、日志平台方。
- **覆盖范围**：全链路追踪分析平台一期全部功能（M1–M17），含数据采集、链路分析、性能监测、业务场景、运营管理、权限审计。
- **不覆盖**：二期能力（成本优化 FinOps、深度 RCA/AIOps、SLO 体系、全链路压测、混沌工程）、深度合规/脱敏（留阶段二）。

```
           │         各中心间专线互通（Kafka 数据同步通道）               │
           │  上海←→苏州A  |  上海←→苏州B  |  苏州A↔苏州B（可选）       │
           └─────────────────────────────────────────────────────────────┘

流量分类：
  ① 用户访问流量：HTTPS → 上海机房1 Nginx → API Gateway → 后端服务 → ES/MySQL
  ② Agent 上报流量：应用进程 → 同机房 Collector(biz直连或dmz代理) → 上海机房1 Kafka
  ③ 跨中心数据流：各中心 Collector → 上海机房1 Kafka（集中存储）
  ④ 内部服务调用：后端服务间通过 RPC / RESTful 通信（同机房内）
  ⑤ 外部接口流量：CMDB/日志平台/监控平台/容器平台的 API 调用
```

**关键约束**：
- **biz 与 dm 区域不通**：dmz 区应用只能经 DMZ 代理上报至 biz 区 Collector；Collector 仅部署在 biz 区。
- **跨中心数据集中**：所有 Collector 数据最终汇聚到**上海机房 1 的 Kafka 集群**。

### 2.8 Redis 存储数据详细描述

#### 2.8.1 数据分类与 Key 规范

| 分类 | Key 模式 | 数据类型 | TTL | 说明 |
|---|---|---|---|---|
| **Agent 心跳与状态** | `agent:heartbeat:{agent_id}` | Hash | 90s | Agent 最后心跳时间、健康状态、版本号、上报速率 |
| **Agent 配置缓存** | `agent:config:{agent_id}` | String(JSON) | 无（主动失效） | 当前生效的采样率/开关/灰度分组等配置快照 |
| **配置下发队列** | `config:push:queue` | List | 24h | 待下发的配置变更事件（灰度/全量） |
| **实时指标缓存** | `metric:realtime:{service}:{instance}:{metric_name}` | String(JSON) | 60s | 最近一次采样的 CPU/内存/JVM/接口 QPS 等实时指标值 |
| **服务健康评分缓存** | `health:score:{service_id}` | Number | 300s | 服务健康评分（算法结果缓存，减少重复计算） |
| **采样率全局配置** | `sampling:global:{datacenter}` | Hash | 无 | 各中心的默认采样率（TPS上限/比率） |
| **限流计数器** | `ratelimit:{ip}:{endpoint}:{minute}` | String(INCR) | 60s | API 请求频率计数（防刷/保护） |
| **用户会话** | `session:{user_id}` | String(JWT/Hash) | 8h | 用户登录会话 Token |
| **权限缓存** | `perm:user:{user_id}` | Set | 10min | 该用户可见的系统/项目 ID 列表（CMDB 权限缓存加速） |
| **拓扑缓存** | `topology:snapshot:{scope}:{time_window}` | String(JSON) | 300s | 拓扑快照（避免每次实时计算） |
| **热点方法缓存** | `hotmethod:{service}:{method}:{window}` | Sorted Set | 600s | 方法耗时排序（用于 TopN 展示） |
| **业务场景映射** | `biz:url:tag:{url_pattern}` | String | 无 | URL → 业务标签映射 |

```
  health:score:svc-payment-001
  sampling:global:shanghai-dc1
  perm:user:user-001
```

| **Kafka metric topic** | 3 天滚动 | 同上 | 已聚合入库后可删 |
| **ES 冷索引** | > 30 天 | ILM Policy → 移至 cold/frozen tier 或归档至对象存储 | 归档后降低 replica 或转存 |
| **ES 超过 1 年数据** | 365 天后 | 降精度归档：Span 详情截断仅保留 header（trace_id/service/duration/status_code） | 保留统计维度，释放空间 |
| **Redis 过期 Key** | 按 TTL 自动过期 | Redis 自身机制 | — |
| **MySQL 审计日志** | 180 天 | 定时任务清理（cron job） | 超期删除 |
| **Agent 断线数据（本地缓冲满）** | 即时丢弃最旧 | Agent 内部环形队列 | 不阻塞业务 |

#### 2.14.2 数据库备份策略
| 数据库 | 备份方式 | 频率 | 保留份数 | RPO 目标 | RTO 目标 |
|---|---|---|---|---|---|
| **MySQL** | mysqldump（逻辑全备）+ Binlog 增量 | 全量每日凌晨 02:00；Binlog 实时同步 | 全量 7 份；Binlog 3 天 | ≤ 1 小时 | ≤ 4 小时 |
| **MySQL** | 从库延迟备份（读从库 dump，不影响主库） | 同上 | 同上 | 同上 | 同上 |
| **Elasticsearch** | Snapshot 至共享存储（NAS/OBS） | 快照每日 03:00 | 30 天 | ≤ 1 小时 | ≤ 2 小时（恢复索引） |
| **Redis** | AOF 持久化 + RDB 定时 | AOF 每秒；RDB 每小时 | RDB 7 天 | ≤ 1 分钟（AOF） | ≤ 30 分钟（RDB 加载） |
| **Kafka** | 多副本机制（replication-factor=2）+ 主题级备份脚本 | 脚本每周 | 4 周 | ≤ 10 分钟 | ≤ 1 小时 |
| **配置文件** | Git 版本化管理（GitLab） | 每次变更自动 commit | 全历史 | — | — |

### 2.15 系统容错设计

| 故障场景 | 容错策略 | 自动恢复 | 影响 |
|---|---|---|---|
| **单个 K8S Pod 崩溃** | HPA 自动扩缩容（最低副本数保障） | K8S RestartPolicy 自动重启 | 无感知（多副本负载均衡） |
| **单个 Node 故障** | Pod 自动迁移至健康 Node（PDB/PodDisruptionBudget 保护） | K8S Scheduler 自动调度 | 短暂抖动（< 30s） |
| **ES 单节点故障** | Replica Shard 自动切换至其他副本（replica=1 时不可用） | ES 自动 rebalance | 查询可能短暂超时（重试可恢复） |
| **MySQL 主库故障** | MHA / Orchestrator 自动故障转移至从库 | 自动提升从库为新主 | 写操作短暂中断（≤ 1 min，RPO≤1h） |
| **Redis 主节点故障** | Sentinel 自动故障转移 | 自动切换到新主 | 少量写入丢失（< 1s） |
| **Kafka Broker 故障** | ISR 机制保证分区可用 | Leader 自动选举 | 生产者短暂阻塞（retries 配置） |
| **Collector 全挂（某中心）** | Agent 本地缓冲不丢数据；恢复后重传 | 重启 Collector | 延迟增加（不影响业务） |
| **上海机房1 整体故障** | （灾难场景）其他中心 Collector 可独立工作，但查询不可用 | 机房级故障需人工介入 | 全局查询不可用；采集侧正常 |
| **biz/dm 网络不通** | dmz 区 Agent 经代理上报（议题 A 方案保障） | 代理高可用（HAProxy 双活） | 代理故障时 dmz 区上报受影响（双代理消除） |
| **跨中心专线中断** | 各中心 Collector 本地缓冲；恢复后批量补传 | 网络恢复后自动 | 延迟增加；标注缺失 |

### 2.19 安全设计

#### 2.22.1 配置类敏感信息必须加密
- **敏感配置项**：数据库密码、Redis 密码、Kafka SASL 凭证、API Key/Secret、第三方系统对接 Token、Agent 采集配置中的敏感字段过滤规则。
- **加密方式**：
  - 存储层：所有敏感配置在 **MySQL**中以 AES-256-GCM 加密存储（密钥由 KMS 或 Vault 管理）。
  - 运行时：应用启动时从配置中心（ConfigMap/环境变量）读取密文，经解密后使用；明文不落地到日志/文件。
  - 传输层：所有组件间通信（gRPC/HTTP）启用 **mTLS**（双向证书）。
- **禁止事项**：配置文件（application.yml/.properties）中不得出现明文密码；Git 仓库中不得提交包含明文凭据的文件。

#### 2.22.2 XSS 安全性拦截
- **前端防护**：
  - 输出渲染时对所有动态内容（服务名、错误码、日志文本）进行 HTML 转义（Vue/React 默认模板自动转义）。
  - 用户输入（搜索框、备注）做白名单校验（仅允许字母数字/常用符号），拒绝 `<script>` / `on*` 等危险字符。
- **后端防护**：
  - API 响应头添加 `X-XSS-Protection: 1; mode=block`、`Content-Security-Policy: default-src 'self'`。
  - 日志内容在返回前端前脱敏处理（详见 2.22.3）。

#### 2.22.3 敏感数据加密
- **链路数据中的敏感字段**：用户身份证号、手机号、银行卡号、密码明文、Token、Session ID 等。
- **处理原则**：
  - **采集时**：Agent 侧配置**敏感字段过滤规则**（正则匹配），命中字段在上报前进行掩码处理（如手机号 `138****1234`），或不上报该 Span attribute。
  - **存储时**：ES 中存储的是已掩码/脱敏后的值；原始敏感数据**不应落入任何存储**。
  - **展示时**：前端二次校验，确保不泄露完整敏感信息。
  - **日志关联**：应用日志中的敏感字段由日志平台自身的脱敏规则处理（本平台负责注入 TraceID，不负责日志内容脱敏）。
- **加密算法**：对称加密 AES-256-GCM 用于存储；哈希 SHA-256 用于比对（如需要）。

#### 2.22.4 异常处理机制
| 异常类型 | 处理策略 | 用户提示 |
|---|---|---|
| 参数非法（空值/越界/格式错） | 输入校验拦截，返回 400 + 明确错误信息 | "参数 xxx 格式不正确，应为..." |
| 权限不足 | RBAC 拦截，返回 403 | "您无权访问该系统数据，请联系管理员" |
| 认证失效（Token 过期/无效） | 返回 401 + 引导重新登录 | "登录已过期，请重新登录" |
| 数据不存在（TraceID 无结果） | 返回 404 + 友好提示 | "未找到匹配的链路数据，请检查 TraceID 是否正确" |
| 外部系统超时（CMDB/日志/监控） | 超时兜底 + 降级提示 | "关联数据加载超时，链路数据仍可查看" |
| 存储查询超时（ES >1s） | 返回超时 + 建议缩小时间范围 | "查询超时，请尝试缩小时间窗口后重试" |
| 服务内部未预期异常 | 全局异常处理器捕获，返回 500 + TraceID | "服务器内部错误，错误编号: ERR-xxx，请联系运维" |
| Agent 断连/离线 | 不阻断业务，标记状态 + 告警 | Agent 台账显示"离线"；链路数据标注"缺失" |
| 跨中心数据缺失 | 不伪造，明确标注 | "跨中心数据缺失（时间窗: ...），当前展示已汇聚部分" |

**全局异常处理架构**：
```
请求 → API Gateway (认证/限流)
  → Controller (参数校验)
    → Service (业务逻辑)
      → DAO (数据访问)
        → External API (外部调用)
          ↓ 异常
GlobalExceptionHandler → {
  400: 参数异常 → 返回友好错误
  401: 未认证 → 返回 401 + 登录引导
  403: 无权限 → 返回 403 + 权限提示
  404: 数据不存在 → 返回 404 + 友好提示
  429: 限流触发 → 返回 429 + "请求过于频繁"
  500: 内部异常 → 记录完整堆栈 + 返回 500 + 错误编号
  502/503/504: 下游不可用 → 降级提示
}
→ 所有异常统一记录到审计日志 + 告警（P0 级立即通知）
```

#### 2.22.5 文件上传要求
- **允许上传场景**（一期有限）：Agent 安装包上传（运营管理）、日志文件导入（调试辅助）。
- **限制**：
  - 文件类型：`.jar` / `.zip` / `.tar.gz` / `.log` / `.json`（白名单校验）。
  - 单文件大小：≤ **100 MB**（Agent 包）。
  - 总存储配额：按项目/租户隔离，默认 **5 GB**（可调）。
  - 扫描：上传后进行**病毒扫描**（ClamAV 集成可选）和**恶意文件检测**（magic number 校验）。
  - 存储：上传至对象存储（MinIO/OBS），不存于本地磁盘或数据库 Blob。
  - URL：生成临时下载链接（有效期 1 小时），过期作废。
  - 审计：记录上传者、文件 hash、时间、IP。

### 3.1 数据采集模块

#### 功能说明
**职责**：负责从被监控应用进程中采集全栈观测数据（链路/指标/日志上下文/运行环境），并通过 OTel Collector 汇聚至后端存储。是整个平台的数据源头。

**输入**：应用进程运行时的调用链路（Span）、JVM 指标、线程状态、接口 QPS/耗时/错误码、资源层（CPU/内存/磁盘/网络）指标、中间件客户端指标（MySQL/Oracle/Redis/Kafka/RabbitMQ）、运行环境元数据（OS/JDK/框架/容器）。

**输出**：统一的 OTel 格式数据（Trace/Metric/Log），写入 Kafka → ES/MySQL/Redis。

**业务规则**：
- Java 应用采用 **SkyWalking Agent** 自动埋点（Web/RPC/SQL/Redis/MQ/运行环境）。
- Go 应用采用 **OTel Agent** 手动埋点。
- 采样率支持两种模式：**最高 TPS 上限**（防雪崩）和**采样比率**（百分比），均为**热配置**（无需重启）。
- 代码级链路（方法逐层明细/堆栈）可通过开关控制开启/关闭。
- Agent 启动参数配置 Collector ip+port；biz 区直连、dmz 区经代理上报（议题 A）。
- 所有数据经 **OTel Collector 定制开发**，统一转换为 OTel 协议格式（SkyWalking 格式 ↔ OTel 格式 ↔ OTel 格式）。
- 容器应用通过 **K8S initContainer** 将 Agent 启动命令注入；VM 应用由运维手动修改启动命令。
- Agent 本地缓冲（环形队列），断连时不阻塞业务线程；缓冲满丢弃最旧数据。
- Agent 向配置中心上报心跳（含版本/健康/速率），形成 Agent 台账。

#### 流程或逻辑实现

```
【应用启动流程】
① K8S 调度 Pod / VM 进程启动
  ↓
② initContainer（K8S）或手动注入（VM）执行：
   java -javaagent:/path/to/skywalking-agent.jar \
        -Dskywalking.collector.backend_service=${COLLECTOR_HOST}:${PORT} \
        -Dskywalking.agent.service_name=${SERVICE_NAME} \
        -Dskywalking.agent.instance_name=${INSTANCE_NAME} \
        -Dskywalking.agent.sample_rate=${SAMPLE_RATE} \
        -jar app.jar
  ↓
③ SkyWalking Agent 初始化：
   - 读取启动参数（Collector 地址、采样率、Service/Instance 名）
   - 注册字节码增强（自动拦截 Web/RPC/SQL/Redis/MQ 调用）
   - 启动后台线程（Metrics 采集、Heartbeat 上报、配置轮询）
  ↓
④ Agent 向配置中心注册（首次握手）：
   POST /api/v1/agent/register
   Body: {agent_id, service, instance, version, dc, zone, ip, port, start_time}
   → 返回 {config: {sample_rate, code_level_enabled, filters}}
  ↓
⑤ Agent 开始采集并按配置上报

【数据采集与上报流程（单次请求）】
① 用户请求进入应用（HTTP/RPC/MQ Consumer）
  ↓
② SkyWalking Agent 拦截（字节码增强）：
   - 创建/传递 TraceID（全局唯一，跨线程/跨 MQ 透传）
   - 创建 Span（记录 operation/start/end/duration/tags/status_code）
   - 采集 JVM 指标（GC/Heap/Thread/Deadlock）
   - 采集接口指标（QPS/耗时/错误码/状态码）
  ↓
③ Agent 判断是否采样（按 TPS 上限 or 采样比率）：
   - 若采样 → 构建 OTel Span → 序列化 → 发送至 Collector
   - 若不采样 → 仅记录本地轻量计数（用于指标聚合）
  ↓
④ Agent 发送数据：
   biz 区：直连 Collector（otlp 端口 4317）
   dmz 区：发送至 DMZ 代理（HAProxy）→ 转发至 biz Collector
  ↓
⑤ OTel Collector 接收：
   Receiver（otlp）接收原始数据
     → Processor（协议转换）：
       - SkyWalking v3 格式 → OTel Span 格式（自定义 processor）
       - Go OTel Agent 数据已是 OTel 格式 → 直接透传
     → Exporter（kafka exporter）→ 写入 Kafka trace-ingest topic
  ↓
⑥ Ingestion Service 从 Kafka 消费：
   - Trace 数据 → 写入 ES（按 trace_id 索引）
   - Metric 数据 → 写入 Redis（实时）+ 聚合后写入 ES（历史）
   - Metadata（Service/Instance/Env）→ 写入 MySQL
  ↓
⑦ Agent 心跳（每 30s）：
   PUT /api/v1/agent/{agent_id}/heartbeat
   → 更新 Redis agent:heartbeat:{id} + MySQL agent_registry 表
```

### 3.2 监控与可视化模块

#### 功能说明
**职责**：提供统一的可观测性视图，包括全局大盘、服务健康评分、多维指标图表、告警一览，以及运营管理所需的用户/项目/集群/字典/审计功能。

**输入**：来自链路查询服务、指标聚合服务、拓扑服务、告警服务的聚合数据。

**输出**：前端可视化页面（图表/表格/拓扑图/火焰图/大盘卡片）。

**业务规则**：
- 统一大盘展示各接入系统的**服务健康总分**，任意视图可下钻至链路/指标/日志详情。
- 服务健康评分综合业务层（QPS/错误率/耗时）、应用层（JVM/CPU/内存）、资源层（容器/主机）指标，基于**可调参数加权算法**计算。
- 指标展示支持服务级/实例级切换，支持时间范围选择（近 1h/6h/24h/7d/30d 自定义）。
- 接口分析与中间件指标支持 **TopN 排行**（吞吐量/错误率/耗时/SQL 耗时）。
- 告警列表携带 **Trace 入口**，一键进入链路视图。
- 运营管理（用户/项目/集群/Agent/字典/审计）作为管理平面嵌入此模块。

#### 流程或逻辑实现

```
【统一大盘加载流程】
① 前端请求 GET /api/v1/dashboard/overview?dc={}&scope={}
  ↓
② 后端聚合查询（并行）：
   a) 从 Redis 缓存读取 `health:score:*` 各服务最新评分（TTL 300s）
   b) 从 ES 聚合查询最近 1h 各服务关键指标（错误率/P99 耗时/QPS）
   c) 从 Redis 读取 `topology:snapshot:global:latest` 拓扑概要
   d) 从 Alert Service 查询活跃告警数量（按级别）
  ↓
③ 计算大盘卡片数据：
   - 总在线服务数 / 异常服务数
   - 平均错误率 / 平均 P99 耗时
   - 活跃告警数（P0/P1/P2）
   - 今日 Trace 总量 vs 昨日同比
  ↓
④ 返回 JSON → 前端 ECharts/D3 渲染

【服务健康评分计算流程】
① 定时任务（每 5 分钟）触发评分计算：
  ↓
② 对每个服务 s，采集三个维度的原始指标：
   - 业务维度 B_s：{error_rate, p99_latency, throughput_trend}
   - 应用维度 A_s：{jvm_gc_pause, thread_deadlock, cpu_usage, memory_usage}
   - 资源维度 R_s：{container_cpu%, container_mem%, disk_usage%, network_iowait}
  ↓
③ 归一化（Min-Max 到 0-100 分）：
   score_dim(s) = 100 × (actual - worst) / (best - worst)
   其中 best/worst 可通过配置参数调整（如 error_rate best=0, worst=5%）
  ↓
④ 加权求和（DD-03 §9.1 权威模型，4 维惩罚制）：
   health = clamp(100 − (E×0.35 + L×0.25 + R×0.20 + A×0.20), 0, 100)
   - 错误率 E（35%）、时延劣化 L（25%）、资源使用率 R（20%）、告警事件 A（20%）
   分级：90~100 健康 / 70~89 关注 / 40~69 警告 / <40 严重
   > ⚠️ 2026-07-20 一致性核查：原 3 维模型（业务40%/应用30%/资源30%，分级 85/70/50）为早期草案，
   > 与 DD-03 §9.1 专家评审 v1（4 维惩罚制，分级 90/70/40）矛盾，此处对齐 DD-03 §9.1 权威基线。
  ↓
⑤ 结果写入 Redis `health:score:{service_id}` 并写入 ES 历史记录
  ↓
⑥ 前端请求时直接读取缓存，无需实时计算

【指标图表查询流程】
① 前端请求 GET /api/v1/metrics/chart?service={}&instance={}&metrics=[]&range={}
  ↓
② 后端判断时间范围：
   - ≤ 1h → 从 Redis `metric:realtime:*` 读取（实时缓存）
   - > 1h → 从 ES 聚合查询（预聚合指标索引）
  ↓
③ 返回时序数据序列 → 前端 ECharts 折线图/面积图渲染

【TopN 排行查询流程】
① 前端请求 GET /api/v1/metrics/topn?service={}&type=latency|error|throughput&n=10&range={}
  ↓
② 后端从 ES 聚合查询：
   - 按 type 字段 bucket 聚合
   - 取 top n
   - 若有 Redis `hotmethod:*` Sorted Set 缓存则优先读取
  ↓
③ 返回排行列表 → 前端表格/柱状图渲染
```

### 3.3 全链路分析模块

#### 功能说明
**职责**：提供端到端调用链路的查询、展示、下钻分析能力，是故障定位的核心工具。

**输入**：用户输入的 TraceID、筛选条件（时间范围/服务/接口/错误码/耗时阈值）。

**输出**：调用链瀑布图（Waterfall）、代码堆栈详情、方法耗时明细、错误/慢调用标记、关联指标/日志入口。

**业务规则**：
- 支持 **TraceID 精确检索**和多条件组合筛选。
- 瀑布图展示整条链路每个 Span 的耗时占比、调用类型、状态码。
- 点击任一 Span 可**下钻至方法级/SQL 级**明细，展示代码堆栈与方法逐层调用耗时。
- 异常（Error Span）自动记录完整堆栈。
- 支持**火焰图**（Flame Graph）展示方法级耗时分布（议题 B 纳入一期）。
- 链路完整性标注：跨中心 Span 缺失标注"链路断裂"；跨中心数据缺失标注"跨中心数据缺失"。
- 瀑布图中可直接点击"查看关联指标"/"查看关联日志"跳转到三域关联视图。

#### 流程或逻辑实现
```
【Trace 检索流程】
① 用户输入 TraceID 或筛选条件提交
  ↓
② 后端 Trace Query Service 解析查询：
   a) 若有 TraceID → 直接查 ES trace-{date} 索引
   b) 若只有筛选条件 → ES bool query 组合过滤：
      - service_name IN [...]
      - duration > threshold (慢调用)
      - status_code IN ['ERROR',...] (错误)
      - span.operation IN [...] (特定接口)
      - timestamp RANGE [start, end]
  ↓
③ ES 返回匹配 Trace 列表（分页，默认 20 条/页）
  ↓
④ 对每条 Trace，构建瀑布图所需 Span 列表（按 parent_span_id 树形排序）
  ↓
⑤ 判断链路完整性：
   - 检查是否有 gap（parent-child 时间差异常 / span 缺失）
   - 检查跨中心 datacenter_id 分布
   - 标注：COMPLETE / PARTIAL_BROKEN / CROSS_DC_MISSING
  ↓
⑥ 返回前端渲染

【Span 下钻流程】
① 用户点击瀑布图中某 Span
  ↓
② 后端查询该 Span 的详细信息：
   a) 基本信息：service/instance/operation/type/duration/tags/status
   b) 若有 code_stack 展示堆栈
   c) 若有 sql_statement 展示 SQL 及耗时
   d) 若有 method_detail 展示方法逐层调用
  ↓
③ 若该服务开启了代码级采集且数据存在：
   - 返回完整堆栈/方法树
   - 否则返回"该服务未开启代码级采集"提示
  ↓
④ 前端渲染：
   - 堆栈面板（折叠树形展示，耗时高亮）
   - SQL 面板（语句 + 执行计划 + 耗时）
   - （可选）火焰图（Flame Graph SVG 渲染）

【火焰图生成流程（新增·议题 B）】
① 用户在 Span 详情页点击"火焰图"
  ↓
② 后端查询该服务在指定时间窗口内的 **Continuous Profiling** 数据
   （若 Agent 开启了 code_level 采样）
  ↓
③ 聚合方法级耗时数据：
   - 按 method_name + package_name 分组
   - sum(duration) as total_time
   - count() as call_count
   - 计算 self_time（排除子调用耗时）
  ↓
④ 构建火焰图数据结构（递归父子关系）
  ↓
⑤ 返回给前端以 D3.js / d3-flame-graph 渲染为交互式 SVG
   - 支持鼠标悬停显示方法名/耗时/占比
   - 支持点击下钻到具体调用
```

#### 页面设计
- **链路检索页**：
  - 顶部：搜索栏（TraceID 精确搜索 / 组合筛选面板）+ 时间范围 + 数据中心切换。
  - 主体：Trace 列表（TraceID / 服务名 / 耗时 / Span 数 / 状态 / 时间 / 操作列[查看]）。
  - 分页控件。
- **链路详情页（瀑布图）**：
  - 顶部面包屑：首页 > 链路检索 > Trace: abc-123。
  - 瀑布图主体：横轴时间刻度，纵向每行为一个 Span，颜色编码（正常=蓝/慢=橙/错误=红/跨中心=紫）。
  - 每个 Span 卡片显示：服务名.操作名 | 耗时(ms) | 状态码 | 点击展开下钻。
  - 右侧面板（点击 Span 后滑出）：基本信息 + 堆栈 + SQL + [关联指标] [关联日志] 按钮。
  - 链路完整性状态标签（完整 / 链路断裂 / 跨中心缺失）。
- **代码堆栈面板**：
  - 折叠树形结构，每层显示：包名.类名.方法名() — 耗时 XXms（占比 YY%）。
  - 异常堆栈红色高亮。
  - [火焰图] 切换按钮。

#### 对应 Redis 设计
| Key | 类型 | TTL | 说明 |
|---|---|---|---|
| `trace:query:result:{trace_id_hash}` | String(JSON) | 300s | Trace 详情查询结果缓存（避免重复 ES 查询） |
| `flamegraph:data:{service}:{span_id}:{window}` | String(JSON) | 600s | 火焰图数据缓存（计算较重） |

#### 对应数据库设计
```sql
-- 链路查询历史（用户常用 Trace 记录）
CREATE TABLE t_trace_query_history (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id         VARCHAR(64)  NOT NULL,
    query_type      ENUM('TRACE_ID','FILTER','RELATED') NOT NULL,
    trace_id        VARCHAR(64)  COMMENT '精确TraceID查询',
    filter_params  TEXT COMMENT '筛选条件JSON',
    result_count    INT DEFAULT 0 COMMENT '命中数量',
    query_time      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_time (user_id, query_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='链路查询历史(保留90天)';
```

> 注：Trace 原始数据存储在 ES（按日期滚动索引 `tracing-span-yyyy-MM-dd`），MySQL 不存 Span 明细。

#### MQ 设计
| Topic | 用途 | 备注 |
|---|---|---|
| `trace-detail-query`（可选） | Trace 详情查询请求排队（高并发时异步化） | 一期规模小可不引入，直接同步查 ES |

#### 安全设计
- TraceID 检索做**频率限制**（同一用户每分钟不超过 60 次），防止滥用。
- 堆栈信息中可能包含代码路径/类名，不属于敏感信息但也不应过度暴露——仅对有权限的用户展示（CMDB 权限校验）。
- 火焰图数据不含业务数据，仅含方法签名与耗时，安全风险低。

### 3.4 应用性能监测模块

#### 功能说明
**职责**：聚焦应用层面的性能分析，包括上下游拓扑、服务/实例/接口级指标、中间件指标、**热点方法分析（议题 B）**、**服务健康评分（议题 B）**。

**输入**：指标聚合服务提供的聚合指标、拓扑服务的依赖关系、链路数据的性能维度提取。

**输出**：上下游拓扑视图、多维指标图表、接口 TopN 排行、热点方法排名、健康评分仪表盘。

**业务规则**：
- **上下游拓扑**：展示服务间的调用关系，点击节点可纵向下钻至实例→容器→主机层资源指标。
- **服务/实例级指标**：JVM（GC/Heap/Thread/死锁）、CPU、内存、线程池、自定义指标图表。
- **接口分析**：吞吐量(QPS)、错误率、API 状态码分布、耗时分布，支持 **TopN 排行**（按吞吐量/错误率/耗时排序）。
- **中间件指标**：MySQL/Oracle/Redis/Kafka/RabbitMQ 客户端指标（吞吐量/耗时/错误率/Slow SQL TopN）。
- **热点方法**：对指定服务方法进行一段时间采集，展示该方法耗时分布（议题 B）。
- **服务健康评分**：综合业务/应用/资源层指标的加权算法评分（可调参数）（议题 B）。

#### 流程或逻辑实现
```
【上下游拓扑加载流程】
① 前端请求 GET /api/v1/topology/upstream-downstream?service={svc}&depth={1|2|3}
  ↓
② Topology Service 查询依赖关系（ES 聚合或 Redis 缓存）：
   - depth=1: 直接上下游（调用方/被调方）
   - depth=2: 二级依赖
   - depth=3: 三级依赖（爆炸半径评估）
  ↓
③ 对每个节点标注：
   - 强依赖（实线红）/ 弱依赖（虚线灰）/ 未知依赖（虚线灰+"?"）
   - 调用次数/平均耗时/错误率（边上标注）
  ↓
④ 返回拓扑数据 → 前端 D3 力导向图渲染
  ↓
⑤ 点击节点事件：
   - 一级点击 → 展开该服务的实例列表
   - 二级点击（实例）→ 展示该实例所在容器/主机信息 + 资源指标
   - 三级点击（主机）→ 展示主机级 CPU/内存/磁盘/网络指标

【热点方法分析流程（新增·议题 B）】
① 用户选择目标服务 + 时间窗口 + 采集时长（如最近 30 分钟）
  ↓
② 后端检查该服务 Agent 是否开启了 code_level 采集：
   - 若开启 → 从 ES 查询该时段的方法级 Span 数据
   - 若未开启 → 返回提示"该服务未开启代码级采集，请联系运维配置"
  ↓
③ 聚合计算：
   - 按 fully_qualified_method_name 分组
   - SUM(duration) as total_duration
   - COUNT(*) as call_count
   - AVG(duration) as avg_duration
   - MAX(duration) as max_duration（慢调用）
   - 排序取 TopN
  ↓
④ 结果写入 Redis `hotmethod:` 缓存 + 返回前端
  ↓
⑤ 前端渲染：
   - 表格：方法名 | 总耗时 | 调用次数 | 平均耗时 | 最大耗时 | 占比%
   - 图表：柱状图（Top10 耗时）
   - 可点击某方法 → 进入该方法的火焰图/调用链详情

【服务健康评分流程（已在 3.2 节详述，此处为模块视角补充）**
- 本模块的健康评分展示页是 3.2 大盘计算的**详细版**：
  - 展示三维得分雷达图（业务/应用/资源）
  - 展示权重配置（可调滑块，仅管理员可改）
  - 展示历史趋势折线图（近 7d/30d 评分变化）
  - 展示扣分明细（哪项指标拖了后腿）
```

#### 页面设计
- **性能监测主页**：
  - 左侧导航：上下游拓扑 / 服务指标 / 接口分析 / 中间件指标 / 热点方法 / 健康评分。
  - 内容区随导航切换。
- **上游��游拓扑**：力导向图，节点可点击下钻，边标注耗时/调用次数/错误率。
- **服务指标页**：
  - 服务/实例切换 Tab。
  - JVM 仪表盘（Heap 用量/ GC 次数/线程数/死锁数）。
  - 资源仪表盘（CPU%/Mem%/Disk%/Net IOWAIT%）。
  - 时间序列折线图（多指标叠加）。
- **接口分析页**：
  - 接口列表表格（接口名/QPS/错误率/P50/P95/P99/状态码分布）。
  - TopN 排行 Tab（按 QPS/错误率/耗时 分别排序）。
- **热点方法页**：
  - 服务选择器 + 时间窗口。
  - TopN 方法耗时表格 + 柱状图。
  - [查看火焰图] [查看调用链] 按钮。
- **健康评分页**：
  - 评分仪表盘（0-100 刻度）。
  - 三维雷达图。
  - 权重配置面板（管理员可见）。
  - 历史趋势折线图。
  - 扣分明细列表。

#### 对应 Redis 设计
（复用 3.2 节相关 Key，此处补充特有 Key）
| Key | 类型 | TTL | 说明 |
|---|---|---|---|
| `hotmethod:{service}:{method}:{window}` | Sorted Set | 600s | 方法耗时排行（Score=总耗时ms） |
| `health:radar:{service_id}` | String(JSON) | 300s | 雷达图三维数据缓存 |

```sql

    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    service_id      VARCHAR(64)  NOT NULL,
    window_start    DATETIME     NOT NULL,
```

### 3.5 业务与基础设施监控模块

#### 功能说明
**职责**：补齐业务视角（URL 业务场景标签）和基础设施视角（主机/容器/网络/中间件的深度监控），衔接应用层监控与底层资源监控。

**输入**：
- 业务侧：用户定义的 URL → 业务场景标签映射关系。
- 基础设施侧：来自统一监控告警平台的主机/虚拟机指标、容器平台指标、中间件指标。

**输出**：
- 业务场景聚合视图（按标签筛选链路与指标）。
- 基础设施指标面板（主机/容器/网络/磁盘层级下钻）。
- 中间件深度指标面板（MySQL/Redis/Kafka/RabbitMQ 客户端视角的性能指标）。

**业务规则**：
- **业务场景设置**：用户为入口 URL 定义业务标签（如 `/api/payment/*` → "支付交易"）。
- **按业务场景聚合**：选择某业务标签后，聚合展示该标签下所有关联链路的关键指标（QPS/错误率/耗时）。
- 基础设施指标从**统一监控告警平台**拉取，在本平台做**关联展示**（不替代原平台，而是打通视图）。
- 容器指标从**容器平台 Prometheus**拉取，关联到对应服务和实例。
- 中间件指标从 Agent 采集的客户端指标获取（吞吐量/耗时/错误率/Slow TopN）。

#### 流程或逻辑实现
```
【业务场景设置流程】
① 管理员/业务负责人进入"业务场景管理"页面
  ↓
② 新增场景：
   POST /api/v1/biz-scenario
   Body: {name:"支付交易", url_pattern:"/api/payment/*",
         description:"", owner:"张三", tags:["核心","资金"]}
  ↓
③ 后端保存到 MySQL t_biz_scenario 表
  ↓
④ 同时更新 Redis `biz:url:tag:{hash(pattern)}` 映射
  ↓
⑤ Agent 在采集 HTTP Span 时，若 request.url 匹配已配置 pattern，
   则在 Span attribute 中附加 biz_scenario="支付交易"

【按业务场景聚合查询流程】
① 用户选择业务场景标签"支付交易" + 时间范围
  ↓
② 后端查询：
   a) 从 Redis 获取该标签对应的 url_pattern 列表
   b) 在 ES 中查询 span.attribute.biz_scenario = "支付交易"
      AND timestamp IN [range]
   c) 聚合：SUM(qps), AVG(latency), ERROR_RATE, TOP_SLOW_TRACES
  ↓
③ 返回聚合数据 → 前端渲染：
   - 场景总览卡片（QPS/错误率/P99 耗时）
   - 关联 Top 慢 Trace 列表
  - [按该场景筛选链路] 按钮 → 跳转链路检索页（自动带入 filter）

【基础设施指标关联流程】
① 用户在服务详情页点击"基础设施"Tab
  ↓
② 后端并行查询：
   a) 从容器平台 API 获取该服务各 Pod 的资源指标（CPU/Mem/Net）
   b) 从统一监控告警 API 获取该服务所在主机的资源指标
  ↓
③ 数据标准化后返回前端渲染：
   - 容器层：Pod 列表 + 每个 Pod 的资源仪表盘
  - 主机层：宿主机列表 + 每台主机的资源仪表盘
  - 网络层：Pod/主机间的网络 I/O 与延迟
```

#### 页面设计
- **业务场景管理**：场景 CRUD 表格 + URL 模式配置 + 标签管理。
- **业务场景看板**：场景选择器 → 该场景下的 QPS/错误率/耗时/Top 慢链路。
- **基础设施 Tab（嵌入服务详情页）**：
  - 子 Tab：容器层 / 主机层 / 网络层 / 中间件层。

### 3.6 日志分析模块

#### 功能说明
**职责**：实现链路与应用日志的关联分析，使排障时可"一条链路看到全部上下文"。

**输入**：Agent 注入到应用日志 MDC 中的 TraceID、日志平台存储的应用日志。

**输出**：在链路详情页中展示**关联的应用日志**，支持按时间窗筛选。

**业务规则**（议题 C 决策）：
- **日志平台 SDK 改造**：不再自行生成 TraceID，改为**复用链路采集 Agent 生成的 TraceID**（Agent 在日志打印时无感注入 MDC）。
- 现状：所有 Java 应用已通过日志 SDK 接入日志平台且存在日志链路；改造后日志链路 ID = Agent TraceID。
- **容器应用注入**：通过 **K8S initContainer** 将 Agent 启动命令（`-javaagent:xxx`）注入到应用启动命令中。
- **VM 应用注入**：由运维**手动修改**应用启动命令添加 `-javaagent` 参数。
- 日志查询时通过 TraceID 从日志平台拉取关联日志条目，在链路详情侧栏展示。

#### 流程或逻辑实现
```
【Agent 注入 TraceID 到日志流程】
① 应用代码执行 logger.info("订单创建完成, orderId=xxx")
  ↓
② SkyWalking Agent 的 Log 插件拦截日志调用：
   - 从当前 TraceContext 获取 trace_id
   - 通过 MDC（Mapped Diagnostic Context）注入：
     MDC.put("traceId", currentTraceId)
     // 可选：MDC.put("spanId", currentSpanId)
  ↓
③ 日志框架（Logback/Log4j2）的 Pattern Layout 中配置
   %X{traceId} → 输出到日志文本
  ↓
④ 日志 SDK（改造后）将该日志（含 traceId 字段）发送到日志平台
  ↓
⑤ 日志平台存储时建立 trace_id 字段的倒排索引

【链路关联日志查询流程】
① 用户在链路详情页点击"查看关联日志"
  ↓
② 前端调用 GET /api/v1/trace/{trace_id}/logs?startTime=&endTime=&limit=50
  ↓
③ 后端向日志平台发起查询请求：
   GET {日志平台Host}/api/v1/logs/search
   Params: traceId={trace_id}, timeRange=[start,end], size=50
  ↓
④ 日志平台返回匹配的日志条目列表
   [{timestamp, level, logger, message, traceId, thread, ...}]
  ↓
⑤ 后端返回给前端 → 侧栏面板渲染日志列表
  ↓
⑥ 用户可点击单条日志展开详情（完整堆栈/上下文）

【日志平台 SDK 改造要点（议题 C 补充）】
改造前（现状）：
  日志SDK 自己生成 requestId/traceId → 形成独立的日志链路

改造后（目标）：
  ① 日志 SDK 检测环境中是否存在 SkyWalking Agent
  ② 若存在 → 从 SkyWalking ContextManager 获取 traceId
  ③ 若不存在 → 降级为自己生成（兼容无 Agent 场景）
  ④ 日志输出时使用获取到的 traceId（优先 Agent 的）
  ⑤ 保证：同一请求的日志 traceId = 链路 traceId（打通）
```

#### 页面设计
- **链路详情页 — 日志侧栏**（右侧抽屉/面板）：
  - 顶部：日志级别筛选（ALL/ERROR/WARN/INFO）+ 时间对齐（相对于链路起点的偏移）。
  - 主体：日志列���（时间戳 | 级别 | Logger | 消息摘要）。
  - 点击展开：完整消息 + 堆栈（若有）。
  - 底部："在日志平台查看更多"链接（跳转日志平台原页面）。
  - 状态指示：日志平台可用/不可用（若超时则提示）。

#### 对应 Redis 设计
（本模块主要依赖日志平台接口，Redis 仅用于缓存查询结果）
| Key | Type | TTL | 说明 |
|---|---|---|---|
| `trace:logs:cache:{trace_id}:{window_hash}` | String(JSON) | 120s | 关联日志查询结果缓存（减少重复调日志平台） |

#### 对应数据库设计
（不在此模块新建表；日志数据存在日志平台，本平台仅做查询桥接。）

#### MQ 设计
（本模块不涉及新的 Topic；日志关联通过 API 实时调用日志平台完成。）

#### 安全设计
- 日志内容可能包含**敏感业务数据**（用户手机号/订单号/金额），前端展示时做**关键字段掩码**（正则匹配手机号/身份证号/卡号 → 替换为 `***`）。
- 日志平台接口调用走**内部网络**（上海机房 1 biz 区内网），不经过公网。
- 日志查询接口做**频率限制**（同 TraceID 每分钟不超过 20 次）。
