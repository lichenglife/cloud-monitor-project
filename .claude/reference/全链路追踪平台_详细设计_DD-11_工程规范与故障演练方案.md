## 1. 编码规范

- **语言/框架**：Java（Spring Boot）/ Go（标准工程结构）；统一格式化（spotless/gofmt）、Lint 卡点。
- **原则落地**：代码评审对照《设计原则与规范》六大维度 + SOLID/AKF，违反可靠性/安全红线且无说明不予合入。
- **防御式编程**：入参校验、外部依赖返回值校验、异常统一处理（错误码 DD-01 §7）。
- **可观测性**：关键路径打点、日志带 `traceId`、禁止明文敏感信息（DD-09/DD-14）。

## 2. 分支模型与提交规范

```
main（生产，受保护）── release/x.y（发布分支）
      ▲                    │
      │                    │
   develop ─── feature/任务号-简述
```

- **分支**：`main`(生产) / `release/x.y`(发布) / `develop`(集成) / `feature/任务号-简述`。
- **提交**：`[任务号] 动作：简述`（如 `[M3] feat: Collector 尾部采样错/慢必采`）；Commit 须关联 TAPD 任务（研发任务拆解 M1–M7）。
- **合入**：PR 须 ≥1 人 Review + CI 全绿（编译/单测/Lint/兼容性测试 DD-13）方可合入 `develop`；`main` 仅经 `release` 合并。
- **发布**：蓝绿 / 金丝雀（DD-13 灰度升级协同），可一键回滚。

## 3. 提测标准

| 标准 | 阈值/要求 |
|---|---|
| 单元测试覆盖率 | 核心逻辑 ≥ 70%，采集/采样/告警 ≥ 80% |
| 接口契约 | 与 DD-01 一致，Mock 约定（前后端并行开发） |
| 静态扫描 | 无高危漏洞、无严重 Lint 违规 |
| 兼容性测试 | 新 Agent 版本入 DD-13 兼容矩阵须通过 |
| 非功能 | 通过 DD-10 压测（P95<1s、背压不丢） |
| 混沌演练 | 通过 DD-11 §5 故障演练用例 |
| 文档 | 接口/数据模型变更同步更新对应 DD |

## 4. 自动化运维

- **CI/CD**：编译 → 单测 → Lint → 镜像构建 → 部署预发；Schema 变更走 Registry 版本校验（DD-14）。
- **配置热更新**：config-center 发布 ≤30s 生效（DD-07 §6），无需重启。
- **发布策略**：金丝雀（1%→10%→100%，DD-13 升级协同）+ 自动回滚（指标异常触发）。

## 5. 故障演练方案（Chaos Engineering）

> 目标：验证采集链路降级与数据缓冲逻辑（设计原则可靠性），确保"存储挂了业务不挂"。

| 用例 | 注入故障 | 预期行为 | 验证点 |
|---|---|---|---|
| C1 | Kafka 断连 | Collector 本地磁盘缓冲、不压垮 Agent | 缓冲有界不 OOM、恢复后重传不丢 |
| C2 | ES Master 故障 | 写入切副本、查询降级缓存/异步 | 查询返回降级视图、写入恢复无丢失 |
| C3 | Collector 实例杀 | 其余实例接管、HPA 扩容 | 采集无感、QPS 自动恢复 |
| C4 | 某中心专线抖动 | 该中心本地缓冲、标注数据延迟 | 其他中心不受影响（DD-08） |
| C5 | Agent CPU 打满 | Agent 自动熔断/降采样 | 业务不受影响、告警触发（DD-13/DD-06） |
| C6 | Redis 不可用 | 配置回源 MySQL、缓存降级 | 功能可用、性能下降可接受 |

- **演练节奏**：预发环境每迭代一次；生产环境季度级（变更窗口）+ 重大版本前。**评审决议 D13=方案A：预发每迭代演练 + 生产季度级（变更窗口 + 审批），重大版本前必做**；不采用方案 B（仅预发，漏生产风险）、方案 C（生产月度，频次过高扰动业务）。
- **演练产出**：故障剧本（GameDay）、恢复 SLA 记录、改进项回流设计（如缓冲容量调优）。
