## 1. 告警总体架构

```
指标/事件源（ES 6 索引 / Redis 缓存 / 平台自监控 metrics）
        │
        ▼
  alert-engine（评估服务，SRP 单一职责）
        │  规则引擎：阈值 / 异常检测 / 静默 / 收敛
        ▼
  npm_online_event（落库，DD-02）
        │
        ▼
统一告警平台投递（Webhook / 消息 / API，对接决议）
        │
        ▼
通知路由（值班 / 群组 / 升级） → 用户（DD-15 告警页）
```

- **评估与投递解耦**：alert-engine 只负责"判定 + 落库 + 投递事件"，投递通道由统一告警平台负责（Y 轴拆分，解耦）。
- **数据来源**：平台内异常经 ES 聚合索引派生（性能/错误/拓扑/健康）；平台自身健康经 DD-05 黄金指标评估（Dogfooding）。

## 2. 告警数据模型（基于 DD-02）

- **规则**：`t_alert_rule`（MySQL，热配置源）+ Redis `alert:silence:{ruleId}`(#6)（静默/收敛态）。
- **事件**：`npm_online_event`（ES，保留 365d），字段见 DD-02 §3.3(6)：`event_id/rule_id/rule_name/severity/alert_type/service_name/instance/title/message/status/fired_time/resolved_time` + 多中心维度。

## 3. 告警规则模型

| 规则要素 | 说明 |
|---|---|
| 评估对象 | service / instance / endpoint / cluster / room / data_center / platform-component |
| 指标源 | `npm_online_performe`(错误率/时延P95/吞吐)、`npm_online_metric`(JVM/CPU)、`npm_online_topology`(边异常)、`npm_online_event`(级联)、DD-05 平台指标 |
| 条件类型 | 阈值（> / < / 区间）、同环比、异常检测（EWMA/环比突增）、缺失（心跳丢失） |
| 持续时间 | `for: 1m/5m`（避免抖动误告） |
| 严重级 | P0 / P1 / P2 / P3（权重: P0=40 P1=20 P2=10 P3=5，呼应 DD-03 §9.1 健康评分） |
| 作用域 | 全局 / 项目 / 集群 / 工程（三级归属，呼应 DD-02） |
| 静默 | 维护期、已知问题窗口（`alert:silence` 缓存，TTL=静默期） |

## 4. 告警分级与响应 SLA

| 级别 | 触发示例 | 通知 | 响应 SLA |
|---|---|---|---|
| P0 | 核心服务错误率>10%、Collector 全阻、跨中心链路中断 | 电话+IM+大屏红闪 | 5min |
| P1 | 错误率>5%、P95 超 SLO 2 倍、Kafka Lag 持续涨 | IM+大屏 | 15min |
| P2 | 单实例异常、告警频次升高、资源>85% | IM | 30min |
| P3 | 提示性（容量预警、弱依赖异常） | 工单 | 次日 |

## 5. 降噪：静默 / 收敛 / 关联

- **静默（Silence）**：维护窗口、已知故障期，规则不产生事件（`alert:silence` 缓存）。
- **收敛（Dedup/Group）**：同一 rule+service+instance 在 `group_window=5m` 内仅产生 1 事件，计数累加；避免告警风暴。
- **关联压缩**：同一根因（如某 DB 故障）触发的多条服务告警，经拓扑（DD-16）向上归因聚合为 1 条"根因告警 + 受影响服务列表"，降低干扰。
- **升级（Escalation）**：P0 未确认超 `ack_timeout=10min` 自动升级至二线 / 值班经理。

## 6. 投递与对接

- **统一告警平台**：alert-engine 经 `alert-events`(Kafka) 推送 → 统一监控告警平台做最终投递/值班/升级（DD-01 §6）。
- **Webhook**：自建通道支持 Webhook（机器人/短信网关），鉴权 HMAC 签名（零信任）。
- **状态闭环**：事件 `status` ∈ {firing, acknowledged, resolved}；恢复事件反向更新 `resolved_time`，避免"已恢复仍红"。
- **接口**：DD-01 §3.8 `/alert/rules`、`/alert/events`、`/alert/silence`、`/alert/rules/{id}/enable`。

## 7. 平台自监控告警（Dogfooding，呼应 DD-05）

- Collector 丢弃率>P95、Kafka Lag 阈值、ES 写入错误、Agent 心跳丢失、平台服务 `/health` degraded → 走同一条 alert-engine 链路，P0 优先。
- 目标：平台自身异常先于业务方被发现（"吃自己狗粮"）。

## 7.5 告警评估引擎与检测延迟（后端架构评审 C6）

- **评估引擎（alert-engine，独立服务 SRP）**：
  - **指标源消费**：alert-engine 订阅预聚合指标流——消费 Kafka `otel-metrics`/聚合结果或定时拉取 ES `npm_online_performe`/`npm_online_metric`（一期推荐**消费预聚合 1m 指标流**，低延迟、不压 ES）。
  - **评估模型**：规则以**滑动窗口**在内存中评估（窗口=规则 `for` 时长）；阈值/同环比/EWMA 命中即生成候选事件，经收敛（§5）后落 `npm_online_event` 并投递。
- **检测延迟（SLA 量化）**：
  - P0 规则：`evaluate.interval=10s`（高频），异常**检出延迟 ≤ 1 个评估周期（~10s）+ 收敛窗口（5m 内仅 1 事件）**；
  - P1/P2/P3 规则：`evaluate.interval=1m`，检出延迟 ≤ 1~2 分钟。
  - "缺失类"（如心跳丢失）：以 `heartbeat_timeout`(60s) + `for` 判定，延迟 ≤ 1~2 分钟。
- **伸缩性**：alert-engine 无状态，按规则/作用域分片并行评估（X 轴 HPA）；评估状态外置（Redis）支持重启续算。

## 8. 设计原则遵循声明

- 可靠性✅（评估/投递解耦、事件落库可回溯、不丢事件） 性能✅（规则评估基于预聚合索引 P95<1s） 可观测性✅（告警本身 Dogfooding、事件带 traceId） 可维护性✅（规则可配置热更新、SRP） 零信任✅（投递鉴权、CMDB 权限过滤可见范围）。
- **待评审 / 已决议**：
  - 1. 各指标默认阈值与 `for` 持续时间的经验值（错误率阈值 5%/10%、P95 超 SLO 倍数）；
  - 2. 收敛 `group_window`(5m) 与升级 `ack_timeout`(10min) 是否合理；
  - ~~3. 异常检测算法选型~~ → **评审决议 D6：引擎=流式消费预聚合流+滑动窗口；算法 ①静态阈值（一期）→ ②同环比/EWMA（二期）→ ③ML（远期）**（详见 DD-06 告警技术决议）；
  - 4. 与统一告警平台对接的字段映射与去重责任边界（本平台去重 or 平台去重）。
