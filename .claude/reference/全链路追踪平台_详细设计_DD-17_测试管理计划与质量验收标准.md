## 1. 文档目的与定位

1. **补齐质量测试文档缺口**：项目原文档集中于架构/设计/工程规范，缺少独立的"质量标准 + 测试管理计划 + 测量指标 + 模块验收标准"聚合文档。
2. **定义可验收的质量门禁**：将抽象的质量目标转化为可度量、可判定（PASS/FAIL）的验收口径，作为提测、发布、试点验收的依据。
3. **驱动部署测试落地**：通过开发一套 **Java 微服务测试铺底组件（npm-trace-testkit）**，在真实环境下持续产生"数据库访问 / Kafka·中间件链路 / 模拟错误 / error 请求 / 日志分析 / 下钻链路"等全场景数据，覆盖所有页面功能，支撑集成、性能、混沌与验收测试。

## 2. 项目质量标准（Quality Standards）

### 2.1 质量模型（四维）

| 维度 | 关注点 | 核心度量 | 达标基线 |
|---|---|---|---|
| **代码质量** | 可维护、可测、无高危 | 单测覆盖率、静态扫描、Lint | 核心≥70%，采集/采样/告警≥80%；无高危漏洞（DD-11 §3） |
| **功能质量** | 需求满足、行为正确 | 需求覆盖率、用例通过率、缺陷密度 | 模块验收标准 100% 覆盖（§6）；上线前阻塞/严重缺陷清零 |
| **非功能质量** | 性能/高可用/容量 | P95、采集不丢率、降级可用率、压测倍数 | P95<1s、采集不丢、试点 3×/上线 10× 压测通过（DD-10） |
| **数据安全质量** | 零信任/脱敏/合规 | 脱敏命中率、越权拦截率、权限兜底 | PII 全脱敏、403 二次校验、CMDB 兜底默认拒绝（DD-09/DD-14） |

### 2.2 质量等级与缺陷分级

沿用《项目管理计划》缺陷分级，作为验收判定与逃逸追责基准：

| 级别 | 定义 | 验收处置 |
|---|---|---|
| 阻塞（Blocker） | 核心链路不可用 / 数据丢失 / 安全越权 | 上线前**必须清零** |
| 严重（Critical） | 主功能失效且无常驻降级 | 上线前**必须清零** |
| 一般（Major） | 功能受限但有可用替代 | 试点前闭环或带风险放行（PM 审批） |
| 轻微（Minor） | 体验/文案/边缘异常 | 跟踪排期修复 |

### 2.3 质量门禁（Gate）

| 门禁 | 触发点 | 准入条件（任一不满足即否决） |
|---|---|---|
| **提测门禁** | 研发 → 测试 | DD-11 §3 全满足：单测覆盖率达标、静态扫描无高危、接口契约一致、非功能压测通过、混沌用例通过 |
| **发布门禁** | 预发 → 生产 | 提测门禁 + 模块验收标准（§6）全 PASS + 缺陷清零（阻塞/严重）+ 压测报告（DD-10 D10）归档 |
| **试点验收门禁** | 试点上线 → 推广 | 发布门禁 + 真实试点系统接入验证 + 采集不丢率 / 查询 P95 实测达标 |

### 2.4 标准基线引用

- **质量标准**（项目管理计划 §3）：Trace 查询 P95 < 1s；采样热配置生效 ≤30s；Agent 无侵入注入；存储周期 1 年。
- **CMMI5 对标**（项目管理计划 §7）：VER（验证）/ VAL（确认）/ QA（质量保证）/ MA（度量分析）过程域由本文档 + DD-11 共同落地。
- **评审决议衔接**：D3（试点 100% 采样）、D5（尾部采样 window=10s/in-flight=20000）、D10（试点 3×/上线 10×）、D13（混沌预发每迭代/生产季度）均为验收硬约束。

## 3. 测试管理计划（Test Management Plan）

### 3.1 测试策略（金字塔 + 专项）

```
         ▲ 验收测试（UAT / 试点）──── 业务可感知价值，少量高价值
        ██ 性能·混沌·安全 专项 ──── 非功能与韧性验证（DD-10/DD-11）
       ███ 端到端 E2E（关键链路）── Trace 查询 / 拓扑下钻 / 三域关联
      ████ 接口/集成测试 ──────── 14 服务契约 + 平台 × 外部平台对接
     ██████ 单元测试 ──────────── 核心逻辑 ≥70%、采集/采样/告警 ≥80%
```

- **左移**：单测/接口契约随代码提交；Mock 依据 DD-01 契约（前后端并行）。
- **右移**：E2E + 性能 + 混沌 + 试点验收，依赖 **npm-trace-testkit** 持续铺底真实流量。
- **自动化优先**：CI 固定跑单测/接口/E2E；性能/混沌设专项门（按需触发）。

### 3.2 测试类型与范围

| 类型 | 范围 | 工具/载体 | 准入文档 |
|---|---|---|---|
| 单元测试 | 采集/采样/告警/聚合/健康评分算法 | JUnit5 + Mockito + JaCoCo | DD-03/DD-07/DD-06 |
| 接口/集成 | 14 服务 REST/gRPC + 外部对接（CMDB/日志/统一监控/容器） | Postman/TestContainers | DD-01 |
| 端到端 E2E | Trace 查询→瀑布→三域、拓扑下钻、告警触发、日志关联 | Playwright（DD-15 §11） | DD-15/DD-16 |
| 性能 | P95<1s、吞吐、背压不丢、压测 3×/10× | JMeter/k6 + ES/Kafka 监控 | DD-10 |
| 混沌 | C1–C6 + 统一告警平台不可达 | Chaos Mesh / 故障注入 | DD-11 §5 |
| 安全 | 越权/脱敏/XSS/传输加密 | ZAP + 人工审计 | DD-09/DD-14 |
| 验收 UAT | 六大模块验收标准（§6）逐条 | npm-trace-testkit + 试点系统 | 本文档 §6 |

### 3.3 测试环境与数据

| 环境 | 用途 | 关键配置 |
|---|---|---|
| 开发自测 | 单测/接口 | 本地 + TestContainers（MySQL/Redis/ES/Kafka） |
| 预发（Staging） | 集成/E2E/性能/混沌 | 全量组件 + npm-trace-testkit 部署（§7） |
| 试点生产 | 试点验收 | 1–2 个真实试点系统接入 + testkit 旁路铺底 |

- **铺底数据来源**：统一由 `npm-trace-testkit`（§7）产生，保证场景可重复、可编排、可回放。
- **脱敏基线**：测试数据 PII 一律走 DD-14 脱敏规则，禁止生产真实敏感数据入测试库。

## 6. 各功能模块验收标准（详细）

> 判定记号：`✅ PASS` = 全部验收点满足；`❌ FAIL` = 任一阻塞/严重项不满足；`⚠️ 带风险放行` = 一般项未闭环但 PM 审批。

### 6.1 模块一：数据采集（M1）

| 验收项 | 验收点 | 验收方法 | PASS 条件 |
|---|---|---|---|
| 协议接入 | OTLP gRPC:4317 / HTTP:4318 接收正常 | testkit 上报 + 抓包核对 | 两类协议均成功入 Collector |
| 采样-头部 | 试点 100% 采样（D3） | 低流量注入校验落库率 | 落库率 100% |
| 采样-尾部 | 错误/慢(>2s) 必采；window=10s/in-flight=20000（D5） | 注入错误/慢请求 + 普通请求 | 错误/慢 100% 采，普通按比例 |
| 自适应采样 | 规则触发调档 ±10%（D4） | 流量/错误率突变观察 | 采样率 ±10% 自适应且无抖动 |
| 幂等去重 | trace_id+span_id 去重 | 重复上报同一 span | 落库无重复 |
| 多中心管道 | biz 直连 / dmz 代理转发；集中上海 DC1（D8/A） | testkit 多实例（biz/dmz） | 两路均入上海 DC1 Kafka |

## 7. 测试铺底组件：Java 微服务（npm-trace-testkit）

> **定位**：为部署测试提供"全场景、可编排、可重复"的真实链路数据，覆盖全部页面功能（§5/§6），是性能/混沌/验收测试的必要使能项。按用户要求新增为开发任务（见《详细设计阶段任务计划》§3.8）。

### 7.1 组件架构

```
npm-trace-testkit（Spring Boot 3 + Java 17，落入 DD-13 兼容矩阵）
├─ scenario-db        # 数据库访问链路（MySQL/JDBC/MyBatis/HikariCP → db span）
├─ scenario-mq        # Kafka 生产消费 + Redis 调用（mq/cache span）
├─ scenario-http      # 下游 HTTP 调用（service span，跨服务串联）
├─ scenario-error     # 模拟错误请求（4xx/5xx 业务异常）
├─ scenario-fault     # error 请求（status=ERROR span，触发尾部采样/告警）
├─ scenario-log       # 日志 SDK（MDC 注入 traceId）+ 多级别日志
├─ scenario-deeptrace # 下钻链路（A→B→C→DB 多层级 / 慢调用 / 超长 Trace / 跨中心）
└─ controller/config  # 场景开关、QPS/错误率/采样标签 可配置、可编排
```

- **链路串联**：经 **SkyWalking Agent 注入**产生 OTLP，span 通过 `traceId` 串联，形成可下钻的瀑布图树。
- **上报通路**：Agent → Collector（biz/dmz 双路）→ Kafka（上海 DC1，D8/A）→ 聚合 → ES 6 索引，与真实接入应用完全一致。
- **可编排**：通过配置文件/接口动态开启/关闭各场景、调节 QPS、错误率、慢调用比例、Trace 深度，支撑回归与压测。

### 7.2 场景 × 页面 覆盖矩阵（"覆盖所有场景"落点）

| # | 测试场景 | 产出的链路/数据 | 覆盖页面（DD-15） | 对应验收项（§6） |
|---|---|---|---|---|
| S1 | 数据库访问 | db span（正常/慢SQL/异常SQL） | 拓扑 / APM / Trace | M1 采样、M2 拓扑(db边)、M4 热点 |
| S2 | Kafka·中间件链路上报 | mq(otel-spans)/cache span + 跨服务调用 | 拓扑 / Trace / 概览 | M1 多中心、M2 强弱依赖、M3 三域 |
| S3 | 模拟错误请求 | 4xx/5xx（业务异常/空指针/参数错） | Trace / 告警 / 概览 | M3 列表(status)、M2 异常高亮、告警 |
| S4 | error 请求 | status=ERROR span | Trace(error 标记) / 告警 | M1 尾部必采、告警准确率 |
| S5 | 日志分析 | 多级别日志 + MDC traceId 注入 | 日志分析 / Trace 关联 | M6 全文/关联/反向定位、M3 三域-Log |
| S6 | 下钻链路分析 | 多层级调用树（A→B→C→DB）/ 慢调用 / 超长 Trace / 跨中心 | 拓扑下钻 / Trace 瀑布 / APM 火焰图 | M2 下钻、M3 瀑布/截断、M4 火焰图 |
| S7 | 资源压力 | JVM/CPU 高占用 | APM 健康评分 / 基建监控 | M4 健康评分(资源20%)、M5 基建 |

> 该矩阵确保**每一类页面功能都能被 testkit 的真实数据驱动验证**，消除"无真实流量难以触发错误/慢链路/跨中心"的测试盲区（T1–T3）。
