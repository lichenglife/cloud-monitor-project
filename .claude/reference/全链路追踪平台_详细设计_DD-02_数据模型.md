## 1. MySQL 元数据模型（功能驱动关系建模）

### 1.0 通用审计字段约定（所有表必含）
| 字段 | 类型 | 说明 |
|---|---|---|
| `created_by` | VARCHAR(64) | 创建人 |
| `created_time` | DATETIME | 创建时间（DEFAULT CURRENT_TIMESTAMP） |
| `updated_by` | VARCHAR(64) | 更新人 |
| `updated_time` | DATETIME | 更新时间（ON UPDATE CURRENT_TIMESTAMP） |

### 1.1 实体关系（核心链路）
```
t_room(机房) ─┬─< t_cluster(集群) ─┬─< t_engineering(工程/应用) ─┬─< t_agent_instance(Agent实例)
             │                    │                              ├─< t_agent_config(Agent配置)
             ├─< t_project(项目) ─┘                              └─< t_agent_registration(注册信息)
             └─< t_engineering / t_agent_instance（room_id 外键）
```
- **项目 → 工程 → Agent实例**：三级归属；工程是 CMDB 应用/系统在本平台的落地单元（对应"5000+ 工程"）。
- **机房主数据**：网络区域（biz/dmz）、机房、集群贯穿项目/集群/Agent，支撑多中心拓扑与 CMDB 权限。

### 1.2 表清单
| 表 | 用途 |
|---|---|
| `t_room` | 网络机房信息（数据中心/城市/网络区域） |
| `t_cluster` | 集群（多中心 biz/dmz） |
| `t_project` | 项目（业务归属） |
| `t_engineering` | 工程/应用（部署单元，关联项目与 Agent） |
| `t_agent_instance` | Agent 实例（运行时） |
| `t_agent_config` | Agent 配置（采样率/最大TPS，热配置源） |
| `t_agent_registration` | Agent 注册实例信息（握手/注册元数据） |
| `t_user` / `t_role` / `t_permission` | 用户/角色/权限 |
| `t_dict` | 系统字典 |
| `t_audit_log` | 审计日志 |
| `t_topology_snapshot` | 拓扑快照 |
| `t_alert_rule` / `t_alert_event` | 告警规则 / 事件 |

### 1.4 设计要点
- **分片/读写分离**：`t_audit_log`、`t_agent_instance`（量大）走读写分离；单表超阈按 `project_id`/`engineering_id` 分表（Z 轴）。
- **配置一致性**：`t_agent_config` 为热配置**源**，经 config-center 发布至 Redis（`agent:config:{clusterId}`），Agent Watch 拉取 ≤30s 生效（呼应 DD-01 / 决议 A）。
- **基数控制**：外键（room_id/cluster_id/project_id/engineering_id）均为低基数，避免高基数属性入表索引。

## 2. Redis Key 设计（12 类，命名/结构/TTL）

| # | Key 模式 | 类型 | TTL | 用途 | 读写路径 |
|---|---|---|---|---|---|
| 1 | `agent:config:{clusterId}` | Hash | 常驻 | Agent 热配置（源自 `t_agent_config`） | config-center 写 / Agent 读 |
| 2 | `service:meta:{serviceName}` | Hash | 30m | 服务元数据（CMDB 同步） | 拓扑计算读 |
| 3 | `topology:cache:{scope}` | String(JSON) | 1m | 拓扑快照缓存 | topology-calc 写 / 查询读 |
| 4 | `trace:dedup:{traceId}` | String | 10m | Span 幂等去重锁 | Collector 写 |
| 5 | `ratelimit:{api}:{key}` | String(INCR) | 1s | 接口限流计数 | API 网关 |
| 6 | `alert:silence:{ruleId}` | Set | 静默期 | 告警静默/收敛 | alert-engine |
| 7 | `user:session:{token}` | Hash | 2h | 用户会话 | user-auth |
| 8 | `dict:cache:{type}` | Hash | 10m | 字典缓存 | 各服务 |
| 9 | `health:score:{serviceName}` | String | 1m | 健康评分缓存 | APM 读 |
| 10 | `metric:rt:{serviceName}` | ZSet | 5m | 实时指标（P95 等） | 监控看板 |
| 11 | `config:version:{clusterId}` | String | 常驻 | 配置版本号（触发热更新） | config-center |
| 12 | `perm:cache:{userId}` | Set | 5m | CMDB 权限缓存（最小权限兜底） | user-auth |
| 13 | `infra:rt:{resourceId}` | ZSet | 5m | 基础设施实时指标（复用多级缓存） | 监控看板 |
| 14 | `infra:cache:{scope}` | String(JSON) | 1m | 基建监控看板缓存 | 监控看板 |

- 内存与淘汰：统一 `allkeys-lru`；热配置常驻不淘汰；避免大 Key（单 Value < 10KB）。
- 多级缓存：本地 Caffeine（进程内）+ Redis（分布式）。

## 3. ES 索引设计（多维监控，6 大索引）

### 3.1 索引总览
| 索引 | 用途 | 数据性质 | 来源 | 粒度 | 保留 |
|---|---|---|---|---|---|
| `trace_npm_jaeger` | 原始 Trace 链路（全量 Span） | 原始 | OTLP Span | 每条 Span | 1 年（ILM） |
| `npm_online_metric` | APM 指标（JVM 等） | 聚合 | Metric + Span 派生 | 1m 桶 | 180d（降采样） |
| `npm_online_db` | DB/中间件调用聚合 | 聚合 | DB/MQ/Cache Span 派生 | 1m 桶 | 180d |
| `npm_online_performe` | 性能聚合（时延/吞吐/错误） | 聚合 | Span 派生 | 1m 桶 | 180d |
| `npm_online_topology` | 调用关系（工程↔工程、工程↔DB/中间件） | 聚合 | Span 派生边 | 1m 快照 | 180d |
| `npm_online_event` | 告警事件 | 事件 | 告警引擎 | 每条事件 | 365d |
| `npm_online_infra` | 基础设施/容器/中间件指标（落库扩展） | 聚合 | 统一监控/容器平台(Prometheus) 派生 | 1m 桶 | 180d |

> 除 `trace_npm_jaeger` 为原始数据外，其余 6 个均为**面向页面查询预计算的不同维度聚合结果**，避免查询时实时重聚合（性能原则：预聚合 + 计算贴近存储）。

### 3.2 聚合管道与监控需求评估
原始 Span 入 `trace_npm_jaeger` 后，经流计算（Collector/Flink）按多维度（service / instance / cluster / room / data_center / network_area / env × time）派生：
- JVM/资源指标 → `npm_online_metric`（APM 页、健康评分输入）
- DB/MQ/Cache Span → `npm_online_db`（DB 监控页：慢查询 / Top 表 / 错误率）
- 时延/吞吐/错误 → `npm_online_performe`（接口 TopN、慢/错分析、Apdex）
- 调用边 → `npm_online_topology`（拓扑图：强弱依赖、异常高亮）
- 告警评估 → `npm_online_event`（告警列表/通知）
- 基建/容器/中间件指标（Prometheus）→ `npm_online_infra`（基建监控页，落库扩展，DD-03 9.5）

**多维度覆盖评估**：6 索引 + 统一维度字段（data_center / network_area / cluster_name / room / env）可支撑"按系统/中心/机房/网络区域/服务/实例/时间"任意切片，满足复杂多维监控。
**待补充评估（已决议）**：基础设施/容器指标经专家评审确定**落库扩展**为 `npm_online_infra`（见 DD-03 9.5），不再仅对接查询，以提升查询效率；实施范围（一期/二期）待排期。

### 3.4 基数与性能控制
- 维度字段全部 `keyword`（低基数）；`attributes` 用 `flattened`；`statement_hash` 降基数；禁止 UserID/OrderID 入索引（DD-03.5）。
- 聚合结果预计算落库，查询仅做轻量检索/简单 term 聚合，保障 P95<1s。
- 分片：trace 初始 6（按写入量扩），聚合索引 3；副本 1。

### 3.5 ILM 与留存
- `trace_npm_jaeger`：**按天 Rollover 索引**（`trace_npm_jaeger-YYYY.MM.DD`），单索引仅承载当日数据，严禁单一 1 年巨型索引；ILM 热(SSD,7d)→温(降副本)→冷(大容量)→超 1 年删除，满足留存。单日索引初始 6 主分片+1 副本（见 §5），按日写入量水平扩容。
- 聚合索引：180d，后期降采样（1m → 5m → 1h）控制成本（Cost 原则）。
- `npm_online_event`：365d。
