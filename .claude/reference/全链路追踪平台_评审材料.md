## 2. 需求基线回顾（评审对齐用）

### 2.1 业务目标
构建"系统内 + 跨系统 + 多中心"的全局链路拓扑与三域关联诊断能力，实现**分钟级故障定位**与**资源利用率可量化提升**。

### 2.2 一期范围（In Scope）
M1 采集（OTel 体系 + Agent 管理 + 全栈指标）、M2 存储、M3 链路检索、M4 全局拓扑、M5 影响分析、M5.1 告警中心、M6 三域关联（含链路关联日志）、M7 多中心汇聚、M14 统一视图/性能监测（**含热点方法/火焰图/健康评分，议题 B 纳入一期**）、M15 权限合规、M16 运营管理平台、M17 业务场景。

### 2.3 不在本期（Out of Scope）
M8 成本优化、M9/M10 中除热点方法/火焰图/健康评分外的其余项（RCA 深度、SLO 体系等）、M11 SLO、M12 压测、M13 混沌、深度合规、全量 200 套接入（一期 10 套/300 工程试点）。

### 2.4 关键量化基线
一期 10 套/300 工程/上海苏州两地三中心、峰值 QPS≈500、采样率 agent 层可控且管理端下发、存储 1 年、查询 <1s、一期 9 个月。

### 2.5 关键架构约束（决策后）
- 两地三中心 + DMZ，各机房互通；同机房 **biz 与 dm 区域网络不通**。
- **部署架构（议题 A 已定）**：各中心 biz 区部署 Collector，各中心 dmz 区部署代理转发至本中心 Collector；所有 Collector 数据传至**上海机房 1 的 Kafka 集群**，跨机房/跨数据中心数据集中存储到上海机房 1 biz Kafka；管理平台与所有 DB（Redis/MySQL/Kafka/ES）部署在**上海机房 1 biz 区**；用户查询经上海机房 1 biz 的 ES/MySQL。

## 3. 关键评审议题（状态与结论摘要）

### 议题 A：biz/dm 网络隔离下的部署架构 ✅ 已决策
- 各中心 **biz 区部署 Collector**；各中心 **dmz 区部署代理**，将 Agent 上报数据转发至本中心 Collector。
- 所有 Collector 数据传至**上海机房 1 Kafka 集群**，跨机房/跨数据中心数据集中存储至上海机房 1 biz Kafka。
- Agent 启动参数配置 Collector 的 ip+port；当前机房应用**直连 Collector（biz 区）或经代理（dmz 区）**上报。
- 管理平台与全部 DB（Redis/MySQL/Kafka/ES）部署在**上海机房 1 biz 区**；链路/监控/告警查询均经上海机房 1 biz 的 ES/MySQL。
- **补充**：该部署方案后续纳入**详细架构方案设计章节**。

### 议题 B：L2 能力范围 ✅ 已决策（纳入一期）
- PRD 3.3 原标注为 L2 的**火焰图、热点方法分析、服务健康评分**，经决策**纳入一期交付**（不再为待确认项）。

### 议题 C：链路关联日志的日志平台接口改造 ✅ 已决策（纳入一期）
- 日志平台 SDK **改造为使用链路采集 Agent 生成的 TraceID**，日志链路接入 TraceID，不再自行生成 TraceID。
- 现状：所有 Java 应用已接入日志 SDK 且存在日志链路，改造为复用 Agent 的 TraceID。
- **接入方式**：容器部署应用利用 **K8S initContainer** 机制将 Agent 启动命令（`java -jar -agent:xxx`）注入；**VM 应用由运维手动修改启动命令**。
- **补充**：需根据 Agent 方案（SkyWalking Agent / OpenTelemetry Agent）制定对接及推广方案，改造工作量需评估。

### 议题 D：存储与查询架构 ✅ 已决策（选型已定，细化待设计）
- 存储选型：**MySQL、Redis、ES、Kafka**，数据**分类分层存储**，满足高可用、查询低延迟、高效率。
- **补充**：容量评估、数据性能、数据架构等**待详细设计**。

### 议题 E：技术选型与 OTel 采集覆盖 ✅ 已决策
- Collector：基于开源 **OpenTelemetry 做基础框架进行定制开发**，以便数据字段扩展与功能扩展。
- Agent：**Java 应用使用 SkyWalking Agent** 采集；**Go 应用使用 OTel Agent**；所有数据上报 Collector 后**按 OTel 协议转换成统一数据格式**。
- **补充**：后续需详细设计方案，明确监控数据报文格式定制与协议转换。
- 前端：**自研**。

## 8. 研发视角：技术依赖 / 数据结构 / 兼容性 / 风险

### 8.1 技术选型（已决策，议题 E/D）
- 采集：Java → **SkyWalking Agent**；Go → **OTel Agent**；日志 MDC 注入 Agent TraceID。
- Collector：**基于 OTel 定制开发**（字段/功能扩展），统一转 OTel 协议。
- 存储：**MySQL + Redis + ES + Kafka** 分层（明细/索引/缓存/队列）。
- 前端：**自研**大盘。
- 部署：各中心 biz Collector + dmz 代理；数据汇聚至**上海机房 1 Kafka/ES/MySQL**（议题 A）。

### 8.2 数据结构（建议）
- Trace/Span 统一格式（经 Collector 转 OTel 协议）：trace_id、span_id、parent、service、instance、operation、span_kind、start/end、duration、status_code、error_code、datacenter_id、zone（biz/dm）、上下游。
- 日志：复用 Agent TraceID 作为关联主键（不再由日志 SDK 生成）。
- ID 规范：全局唯一 TraceID、datacenter_id/zone 编码统一。

### 8.3 兼容性与版本
- Java（SpringBoot 2.x/3.x）经 SkyWalking Agent；Go 经 OTel Agent。
- K8S 1.21/1.25 initContainer 注入；VM 手动注入。
- 中间件（MySQL/Oracle/Redis/Kafka/RabbitMQ/Tomcat/BES）埋点适配。

### 8.4 实现风险
- biz/dm 隔离、跨中心汇聚（议题 A 方案已定，待细化）。
- 多中心一致性、时钟同步。
- 采集开销控制（异步/缓冲、采样率双模式、限流）。
- 存储成本与查询性能（议题 D 选型已定，待容量设计）。

## 9. 评审决策详述（已定）

### 9.1 议题 A — 部署架构
| 决策点 | 结论 |
|---|---|
| 各中心 Collector 部署 | 各中心 **biz 区**部署 Collector |
| dmz 区处理 | 各中心 **dmz 区部署代理**，将 Agent 数据转发至本中心 Collector |
| 跨中心汇聚 | 所有 Collector 数据传至**上海机房 1 Kafka 集群**，跨机房/跨数据中心数据集中存储至上海机房 1 biz Kafka |
| Agent 上报 | 启动参数配置 Collector ip+port；同机房应用**直连 Collector（biz）或经代理（dmz）**上报 |
| 平台与 DB 部署 | 管理平台 + 全部 DB（Redis/MySQL/Kafka/ES）部署在**上海机房 1 biz 区** |
| 查询路径 | 用户经上海机房 1 biz 的 ES/MySQL 查询链路/监控/告警数据 |
| 补充 | 该方案纳入**详细架构方案设计章节** |

### 9.2 议题 B — L2 范围
- **火焰图、热点方法分析、服务健康评分纳入一期交付**（PRD 3.3 原 L2 标注作废，M14 相关需求转为一期）。

### 9.3 议题 C — 链路关联日志
- 日志平台 SDK 改造为**复用 Agent 生成的 TraceID**（不再自行生成）。
- 现状 Java 应用已接日志 SDK，改造为接入 Agent TraceID。
- 注入方式：容器用 **K8S initContainer** 注入 `java -jar -agent:xxx`；VM 由运维手动改启动命令。
- 补充：需据 Agent 方案（SkyWalking/OTel）制定对接推广方案，**改造工作量待评估**。

### 9.4 议题 D — 存储架构
- 选型：**MySQL + Redis + ES + Kafka**，分类分层存储，满足高可用/低延迟/高效率。
- 补充：**容量评估、数据性能、数据架构待详细设计**。

### 9.5 议题 E — 技术选型
- Collector：基于开源 **OTel 定制开发**（字段/功能扩展）。
- Agent：**Java=SkyWalking Agent；Go=OTel Agent**；上报后经 Collector **按 OTel 协议转统一格式**。
- 补充：**监控数据报文格式定制与协议转换待详细设计**。
