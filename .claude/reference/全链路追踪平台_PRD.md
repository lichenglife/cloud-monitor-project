## 2. 背景（Background）

### 2.1 行业背景
可观测性已成为大规模微服务运维的核心基础设施，由**指标、日志、追踪、拓扑**四大支柱构成。Google Dapper 奠定分布式追踪方法论，阿里鹰眼实现调用堆栈还原与强弱依赖识别，字节跳动构建亿级 QPS 全栈可观测体系，美团 CAT/MTrace 实现秒级全量采集与多语言支持。行业趋势是从"三套系统各自为战"走向"统一采集 + 后端关联分析"。

### 2.2 公司现状与痛点
- **链路层空白**：当前仅有基础设施监控、容器层监控、日志，**无应用链路层（Trace）监控**。
- **数据孤岛**：指标、日志、链路分属不同自研平台，缺乏统一关联主键。
- **拓扑缺失/不准**：缺乏统一拓扑，依赖关系靠人工梳理。
- **排障低效**：故障定位跨多平台跳转，上下游影响分析依赖经验。
- **资源视角割裂**：链路与资源指标不关联。

### 2.3 现有平台（对接基础）
| 平台 | 能力 | 接口 |
|---|---|---|
| CMDB | 工程/系统名、应用/运维负责人、权限、机房、数据中心 | 元数据查询 |
| 日志平台 | 各系统已通过日志 SDK 对接（Java 应用已接入，存在日志链路），提供日志查询 | 日志查询 API（**需改造复用 Agent TraceID**） |
| 统一监控告警平台 | 物理机/虚拟机基础设施层 + 中间件/数据库监控 | 指标/告警接口 |
| 容器平台 | K8S 容器化部署；Prometheus/Grafana 容器资源监控 | 容器指标/部署信息 |

### 2.4 关键架构约束（含评审决策）
- **两地三中心 + DMZ**：上海、苏州两地，三个数据中心；各机房网络互通。
- **同机房区域隔离**：同一机房内 **biz 区域与 dm 区域网络不通**。
- **混合部署**：K8S（1.21/1.25）容器化为主 + 部分虚拟机；中间件含 MySQL/MongoDB/Oracle/Redis/Kafka/RabbitMQ/MQ/Tomcat/宝蓝德(BES)、nginx。
- **部署架构（评审议题 A 已定）**：
  - 各中心 **biz 区部署 Collector**；各中心 **dmz 区部署代理**，将 Agent 上报数据转发至本中心 Collector。
  - 所有 Collector 数据传至**上海机房 1 的 Kafka 集群**，跨机房/跨数据中心数据**集中存储至上海机房 1 biz Kafka**。
  - Agent 启动参数配置 Collector 的 ip+port；同机房应用**直连 Collector（biz）或经代理（dmz）**上报。
  - 全链路追踪平台**管理平台与全部 DB（Redis/MySQL/Kafka/ES）部署在上海机房 1 biz 区**；用户查询经上海机房 1 biz 的 ES/MySQL。
  - 该部署方案后续纳入**详细架构方案设计章节**。

## 6. 功能需求详述（EARS）

> 标注：**U**=Ubiquitous、**E**=Event-driven、**W**=Unwanted、**S**=State-driven、**O**=Optional。

### 6.1 M1 统一数据采集层（Java=SkyWalking / Go=OTel + 定制 Collector）

**U（系统始终）**
- REQ-M1-U1：系统应始终为每一个请求生成并传递**全局唯一 TraceID**，跨进程/线程/消息队列透传。
- REQ-M1-U2：系统应始终在采集数据中携带 **datacenter_id** 与 **zone（biz/dm）**。
- REQ-M1-U3：系统应始终对业务保持低侵入；**Java 服务采用 SkyWalking Agent 自动埋点，Go 服务采用 OTel Agent 手动埋点**。

**E（事件触发）**
- REQ-M1-E1：当 **K8S 应用经 initContainer 注入 Agent 启动命令（`java -jar -agent:xxx`）**，或 **VM 应用由运维手动注入启动命令**，且进程启动被注入 Agent 时，系统应自动开始采集。
- REQ-M1-E2：当管理端下发新采样率配置时，Agent 应 **≤30 秒** 生效且**无需重启业务进程**。
- REQ-M1-E3：当应用经消息队列收发消息时，系统应自动注入/提取 Trace 上下文，保证异步链路连续。
- REQ-M1-E4：当管理端切换"代码级调用链采集开关"时，Agent 应相应开启/关闭方法级明细与堆栈采集。
- REQ-M1-E5：当 Agent 启动时，系统应读取启动参数中配置的 **Collector ip+port**，并按同机房**直连 Collector（biz）或经代理（dmz）**完成上报。

**W（异常/故障）**
- REQ-M1-W1：若 Agent 与 Collector 网络中断，Agent 应在本地缓冲（环形队列，上限可配）暂存，恢复后重传；缓冲满丢弃最旧数据且不阻塞业务线程。
- REQ-M1-W2：若 Agent 自身异常（OOM/崩溃），业务进程应不受影响，并上报 Agent 健康异常事件。
- REQ-M1-W3：若实例上报速率超阈值（防雪崩），Collector 应限流丢弃超额数据并记录限流事件。

**S（状态相关）**
- REQ-M1-S1：当 Agent 处于"暂停采集"状态时，停止上报但仍维持心跳。
- REQ-M1-S2：当处于配置灰度发布状态时，仅命中灰度分组的 Agent 应用新配置。

**O（可选）**
- REQ-M1-O1：在启用 eBPF 无侵入采集时，系统应能在内核态补充网络层调用与延迟（后续 PoC）。

**补充采集维度**
- REQ-M1-APP1（应用层）：采集服务/实例维度 JVM（GC）、线程数、死锁线程；服务/实例/接口级请求数、耗时、错误数、状态码（可指定）。
- REQ-M1-RES1（资源层）：采集主机/服务/实例 CPU、内存、磁盘、网络。
- REQ-M1-MID1（中间件）：采集 MySQL/Oracle/Redis/Kafka/RabbitMQ 等客户端吞吐量、耗时、错误率。
- REQ-M1-ENV1（运行环境）：采集 OS、JDK、开发框架、Tomcat 容器信息。
- REQ-M1-SMP1（采样率双模式）：支持**最高 TPS 上限**与**采样比率**两种热配置。
- REQ-M1-COLL1（Collector）：**Collector 应基于 OpenTelemetry 定制开发**，所有上报数据按 **OTel 协议转换成统一数据格式**（Java/SkyWalking 与 Go/OTel 数据统一）。

### 6.2 M2 统一接入与存储（MySQL+Redis+ES+Kafka 分层）
- REQ-M2-U1：系统应始终支持 Trace/Metric/Log 三类数据统一接入与关联存储；存储采用 **MySQL（业务/元数据）+ Redis（缓存）+ ES（检索/明细）+ Kafka（管道/队列）** 分层架构，存储周期 **≥1 年**。
- REQ-M2-E1：当收到链路查询请求时，系统应在 **<1 秒（P95）** 内返回单条 Trace 详情（一期规模下）。
- REQ-M2-E2：当写入高峰（峰值 QPS≈500）时，系统应通过分层与冷热策略保障性能。
- REQ-M2-W1：若存储节点不可用，写入应快速失败并告警，不阻塞采集侧（Agent 缓冲）。
- REQ-M2-W2：若查询延迟 >1s，应记录慢查询并触发容量预警。

### 6.3 M3 链路追踪与检索
- REQ-M3-U1：系统应始终支持 **TraceID 精确检索** 完整调用链，并**瀑布图展示端到端链路埋点与请求耗时**。
- REQ-M3-U2：系统应始终记录耗时、返回码、错误码、调用方/被调方服务名。
- REQ-M3-E1：点击某 Span 时，下钻展示**方法级/SQL 级**明细与耗时占比。
- REQ-M3-E2：点击"代码栈"时，展示**代码堆栈、方法耗时**；**异常时自动记录堆栈**。
- REQ-M3-E3：按"错误码/慢调用"筛选时，列出符合条件 Trace 并支持排序。
- REQ-M3-W1：若 Trace 因跨中心丢失部分 Span，标注"链路断裂"缺口，不静默丢弃。
- REQ-M3-S1：某服务"异常"时，链路列表高亮其相关 Trace。

### 6.4 M4 全局链路拓扑（含强弱依赖与纵向下钻）
- REQ-M4-U1：系统应始终基于真实调用链**自动发现**依赖，不依赖人工录入。
- REQ-M4-U2：系统应始终将 CMDB 元数据（机房/系统/负责人）**补全**到拓扑节点与边。
- REQ-M4-E1：识别到 A→B **强依赖**（B 故障致 A 不可用）时实线/红标；弱依赖（可降级）虚线/灰标。
- REQ-M4-E2：点击服务节点跳转该服务实例依赖拓扑，子页面展示实例所在**容器、主机信息**。
- REQ-M4-E3：查看上下游拓扑时展示关联关系，点击节点**逐层下钻至主机层资源指标**。
- REQ-M4-W1：依赖数据缺失无法判定强弱时，默认标注"未知依赖"并提示人工确认。
- REQ-M4-S1：实时模式展示近期调用；全量模式展示历史累计。

### 6.5 M5 故障影响分析
- REQ-M5-E1：某服务被标记故障/告警时，沿拓扑**自动计算上下游影响面**（直接/间接），输出爆炸半径。
- REQ-M5-W1：影响面因拓扑不完整有盲区时，提示"影响分析可能存在遗漏"，不绝对结论。

### 6.6 M5.1 统一告警中心
- REQ-M5.1-U1：始终支持基于 Trace/Metric 告警规则并统一收敛至告警中心，且**携带关联 Trace 入口**。
- REQ-M5.1-E1：指标偏离**动态基线**时触发告警（静态阈值并行可选）。
- REQ-M5.1-E2：同根因短时间多告警时按服务/根因**降噪收敛**为单条并通知 Oncall。
- REQ-M5.1-W1：通知渠道发送失败应重试并标记"通知失败"。

### 6.7 M6 三域关联诊断（链路关联日志改造）
- REQ-M6-U1：系统应始终以 **TraceID 为关联主键**联动链路、指标、日志三方数据。
- REQ-M6-E1：链路详情点击"查看关联指标/日志"时，同上下文展示该 Trace 时间窗内指标曲线与日志。
- REQ-M6-E2（链路关联应用日志）：**日志平台 SDK 应改造为复用链路采集 Agent 生成的 TraceID**（不再自行生成），Agent 在应用打印日志时**无感注入 TraceID 到日志上下文（MDC）**；日志平台保存后，用户可在链路详情**按 TraceID 查询关联应用日志**。容器应用经 K8S initContainer 注入 Agent；VM 应用由运维手动注入。
- REQ-M6-W1：关联平台接口超时或无数据时标注"关联数据暂不可用"，不影响链路本身。

### 6.8 M7 多中心链路汇聚（数据集中至上海机房 1）
- REQ-M7-U1：系统应始终为跨中心调用保留 **datacenter_id** 与全局 TraceID；各中心 Collector 数据汇聚至**上海机房 1 Kafka 集群**，跨机房/跨数据中心数据**集中存储至上海机房 1 biz Kafka/ES**，用户查询经上海机房 1 biz 的 ES/MySQL。
- REQ-M7-E1：当同一 TraceID 出现在多个中心时，汇聚层应在查询时**跨中心拼接**为完整调用链并按因果/时间排序。
- REQ-M7-W1：若某中心未上报或网络中断导致部分数据暂不可达，系统应展示已汇聚部分并标注"跨中心数据缺失（时间窗）"，不伪造完整链路。
- REQ-M7-W2：若跨中心时钟不同步导致时序错乱，应基于因果关系重排，偏差超阈值时告警。
- REQ-M7-S1：当某中心处于"隔离（biz/dm 不通）"状态时，该中心 Agent 仅经由允许通道（dmz 代理）上报。

### 6.9 M14 统一可观测视图与性能监测（含议题 B 纳入一期项）
- REQ-M14-U1：始终提供统一大盘，汇总各接入系统链路健康总分，任意视图可**下钻**至链路/指标/日志详情。
- REQ-M14-E1：点击某服务健康分异常时联动跳转至该服务拓扑/链路/告警视图。
- REQ-M14-PERF1：页面支持**服务级/实例级** JVM、CPU、内存、线程池指标图表。
- REQ-M14-PERF2：页面支持**服务级/实例级**接口吞吐量、错误率、API 状态码分布、耗时图表，并按吞吐量/错误率/耗时 **topN 排名**。
- REQ-M14-PERF3：页面支持**服务级/实例级**关联中间件客户端指标（如 SQL 错误率、耗时 topN）。
- REQ-M14-PERF4（热点方法分析，一期·议题 B）：支持对指定服务方法进行一段时间采集，展示该时段方法耗时。
- REQ-M14-PERF5（火焰图，一期·议题 B）：支持方法级**火焰图**展示。
- REQ-M14-PERF6（服务健康评分，一期·议题 B）：综合业务、应用、资源层指标诊断，基于**可调参数算法**计算服务健康度评分。

### 6.10 M15 权限与合规（基础）
- REQ-M15-U1：始终基于 **CMDB 权限信息**控制用户对系统链路数据可见范围。
- REQ-M15-U2：始终记录关键操作（配置变更、查询、导出）审计日志。
- REQ-M15-E1：用户访问非权限范围内系统数据时拒绝并提示无权限。
- REQ-M15-W1：CMDB 权限数据不可用时默认**最小权限**拒绝并告警。
- REQ-M15-O1：启用敏感字段脱敏（阶段二）时按规则脱敏。

### 6.11 M16 运营管理平台
- REQ-M16-U1（用户管理）：始终支持**用户创建、登录、登出、注销**，内置**超级管理员**。
- REQ-M16-U2（项目管理）：始终支持**创建项目**、清单展示、项目与集群关联。
- REQ-M16-U3（集群管理）：始终展示 **K8S 集群列表**（网络区域、机房、集群名）。

- REQ-M16-U5（系统管理）：始终支持**配置字典**维护。
- REQ-M16-U6（操作审计）：始终支持查看**用户所有操作审计**。

## 7. 关键流程说明（Flows）

### 7.1 数据采集与上报流程
应用进程（K8S initContainer / VM 手动注入 Agent）→ SkyWalking(Java)/OTel(Go) 自动埋点生成 Span + TraceID（跨 MQ 透传、日志 MDC 注入 Agent TraceID）→ 全栈指标采集 → 本地缓冲 → 上报本中心 Collector（biz 直连 / dmz 代理）→ **所有 Collector 数据汇聚至上海机房 1 Kafka** → 存储（ES/MySQL/Redis 分层）。

### 7.2 链路追踪查询流程
用户输入 TraceID / 筛选 → 查询路由至**上海机房 1 ES/MySQL** → 返回瀑布图 → 点击 Span 下钻方法/SQL/代码栈 → 点击"关联"联动指标/日志（含按 TraceID 查应用日志）。

### 7.3 故障定位流程
告警 → 进入关联 Trace → 定位错误/慢 Span（代码栈/火焰图辅助）→ 沿 M4 拓扑看上下游影响面 → 联合指标/日志确认根因。

### 7.4 多中心链路汇聚流程
各中心独立采集，Collector 上报至**上海机房 1 Kafka** 集中存储 → 查询时凭全局 TraceID + datacenter_id 跨中心拼接 → 因果/时间排序；biz/dm 隔离下仅经 dmz 代理上报。

### 7.5 Agent 配置下发流程
管理端修改采样率（最高 TPS/比率）/代码级开关 → 配置中心校验 → 灰度命中 → Agent 拉取/推送 → ≤30s 生效（无重启）→ 心跳回报。

### 7.6 运营管理流程
管理员登录（超级管理员）→ 创建用户/项目、绑定集群 → Agent 注册进台账 → 配置字典维护 → 操作留痕审计。

## 9. 数据指标与埋点要求（Metrics & Instrumentation）

### 9.1 链路采集字段（Span 统一格式，经 Collector 转 OTel）
TraceID、SpanID、ParentSpanID、服务名、实例、接口/方法名、调用类型（HTTP/RPC/MQ/SQL）、开始/结束时间、耗时、返回码、错误码、datacenter_id、zone（biz/dm）、上下游服务。

### 9.2 指标采集字段
- 应用层：JVM（GC/堆/非堆）、线程数、死锁线程；服务/实例/接口级：请求数、耗时、错误数、状态码（可指定）。
- 资源层：主机/服务/实例 CPU、内存、磁盘、网络。
- 中间件：MySQL/Oracle/Redis/Kafka/RabbitMQ 客户端吞吐量、耗时、错误率。
- 运行环境：OS、JDK、开发框架、Tomcat。

### 9.3 埋点要求
- Java（SpringBoot/SpringCloud）：**SkyWalking Agent** 自动埋点（Web/RPC/SQL/Redis/MQ/运行环境）；Go：**OTel Agent** 手动埋点。
- 跨线程/跨 MQ 上下文透传；日志 MDC 注入 Agent TraceID（复用，不再自生成）。
- 采样：Agent 层可控，管理端下发；支持**最高 TPS / 采样比率**双模式；代码级链路开关可配。
- 注入：K8S initContainer 注入 `java -jar -agent:xxx`；VM 手动。

### 9.4 关键平台 SLA
| 指标 | 目标 |
|---|---|
| Trace 查询延迟（P95） | < 1s（经上海机房 1 ES） |
| Trace 存储周期 | ≥ 1 年 |
| Agent 配置生效时延 | ≤ 30s（无重启） |
| 采集对业务线程阻塞 | 零阻塞（异步/缓冲） |
| 数据完整性（跨中心拼接） | 可标注缺失，不伪造 |

## 11. 边界与异常场景（Boundary & Exception）

| 场景 | 预期处理 |
|---|---|
| biz 与 dm 区域不通 | Agent 经 dmz 代理上报；汇聚层不越权；拓扑标注隔离边界 |
| 跨中心网络中断 | 展示已汇聚部分（上海机房1）并标注"跨中心数据缺失"，不伪造 |
| 跨中心时钟不同步 | 基于因果重排，偏差超阈值告警 |
| Agent 与 Collector 断连 | 本地缓冲重传；缓冲满丢旧数据且不阻塞业务 |
| 链路断裂（Span 缺失） | 标注"链路断裂"缺口 |
| 存储不可用 | 快速失败并告警，Agent 缓冲兜底 |
| CMDB 权限异常 | 最小权限拒绝并告警 |
| 关联平台接口超时 | 标注"关联数据暂不可用" |
| Agent 离线/异常 | 台账标识并提示 |
