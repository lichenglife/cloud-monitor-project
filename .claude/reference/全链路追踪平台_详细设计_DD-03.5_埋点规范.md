## 1. Span 命名规范

### 1.1 命名格式
- 统一采用 **`{组件类型}.{操作}`** 语义化命名，避免将业务参数写入名称。
- 推荐（自动埋点由 Agent 生成）：
  - `http.server.request`（服务端入站 HTTP）
  - `http.client.request`（出站 HTTP 调用）
  - `mysql.query` / `redis.command` / `kafka.produce` / `kafka.consume`
  - `rpc.server` / `rpc.client`（Dubbo/gRPC）
- 禁止：将 UserID、OrderID、随机 ID 拼入 Span 名称（导致高基数、索引爆炸）。

### 1.2 Span Kind
- `SPAN_KIND_INTERNAL` / `SERVER` / `CLIENT` / `PRODUCER` / `CONSUMER`，由 Agent 按调用方向自动判定。

## 2. 标准 Tag / Attribute 键名（语义约定）

### 2.1 通用（必填/推荐）
| Key | 含义 | 基数要求 | 是否索引 |
|---|---|---|---|
| `service.name` | 服务名（CMDB 应用名） | 低（枚举） | 是 |
| `service.instance.id` | 实例标识（Pod/IP） | 中 | 是 |
| `service.version` | 版本 | 低 | 否 |
| `deployment.env` | 环境（prod/test） | 低 | 是 |
| `host.name` / `host.ip` | 主机 | 中 | 是 |
| `k8s.cluster.name` / `k8s.namespace.name` / `k8s.pod.name` | K8s 拓扑 | 低/中/中 | 是 |
| `data.center` / `computer.room` | 数据中心/机房（多中心标识） | 低 | 是 |
| `trace.id` / `span.id` / `parent.span.id` | 链路标识 | — | 是 |
| `span.kind` | 调用方向 | 低 | 是 |
| `status.code` | 0=OK / 1=ERROR / 2=UNSET | 低 | 是 |
| `status.message` | 错误描述 | 高（不索引） | 否 |
| `error` | 是否错误（true/false） | 低 | 是 |

### 2.2 HTTP
| Key | 含义 | 基数要求 |
|---|---|---|
| `http.method` | GET/POST… | 低（索引） |
| `http.url` | 完整 URL（含参数时须脱敏） | 高（不索引/仅采样存储） |
| `http.route` | 路由模板 `/api/v1/order/:id` | 低（索引） |
| `http.status_code` | 200/404/500… | 低（索引） |
| `http.client_ip` | 客户端 IP | 中（脱敏后索引） |

### 2.3 数据库 / 缓存
| Key | 含义 | 基数要求 |
|---|---|---|
| `db.system` | mysql/redis/mongodb… | 低（索引） |
| `db.name` | 库名 | 低 |
| `db.operation` | select/insert/update… | 低（索引） |
| `db.sql.table` | 表名 | 低（索引） |
| `db.statement` | SQL 语句（参数脱敏） | 高（不索引，仅采样留存） |

### 2.4 消息队列
| Key | 含义 | 基数要求 |
|---|---|---|
| `messaging.system` | kafka/rocketmq… | 低（索引） |
| `messaging.destination` | topic/queue | 低（索引） |
| `messaging.operation` | produce/consume | 低（索引） |

### 2.5 分级约定
- **强制（MUST）**：`service.name`、`span.kind`、`status.code`、`trace.id`、`span.id`、各组件 `system` 类键。
- **推荐（SHOULD）**：`http.route`、`db.operation`、`db.sql.table`、`k8s.*`、`data.center`。
- **可选（MAY）**：业务自定义 Attribute（放入 Attribute，不做索引，避免基数膨胀）。

## 3. 基数与脱敏控制（红线）
- Tag 的 **Key 总数受限**（单 Span ≤ 30 个索引 Tag）；Value 必须为枚举或低基数。
- **禁止**：UserID、OrderID、手机号、身份证号、明文 token 作为 Tag 或索引字段；如需关联，写入 Attribute 或日志，且不索引。
- 自动脱敏：采集层（Collector/Agent）对 `db.statement`、`http.url`、自定义Attribute 中命中的 PII 正则（手机号/身份证/密码）做掩码（如 `138****1234`）。
- 与 **DD-14 数据治理与成本治理方案** 的脱敏 ETL 规则保持一致。

## 4. 版本化与向后兼容
- 本规范版本 `v1`；新增键名走"新增"语义，禁止删改已发布键名（违背向后兼容原则）。
- Agent 与 Collector 按 Schema 版本协商；旧版 Span 字段缺失时以默认值兼容。

## 5. 与采集/采样的关系
- 头部采样在 Agent 侧依 `http.status_code`/`db.operation` 等低基数键决策；尾部采样在 Collector 侧依完整 Span 决策（错误/慢请求必采）。
- 高基数 Attribute 不参与采样决策，仅随采样结果留存。

## 6. 设计原则遵循声明
- 控制基数✅ 资源消耗红线✅（规范约束降低 Agent/ES 负担） 采样即正义✅ 向后兼容✅ 防御式编程✅（脱敏校验） 最终一致性✅（命名约定异步生效）
- 待评审确认项：Tag Key 上限具体数值、自定义 Attribute 上报白名单机制。
