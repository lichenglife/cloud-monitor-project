## 一、六大核心设计维度

### 1. 可靠性与容错性（Reliability & Resilience）
- **面向失败设计**：假设任何组件都可能失效（网络断、磁盘坏、进程崩）。Collector 须具备背压机制；ES 宕机时不得拖垮业务服务（采集链路与存储链路解耦、缓冲+降级）。
- **降级与熔断**：非核心功能高压下让路。存储压力大时优先保采集，允许丢弃非错误链路 Trace。
- **幂等性**：Kafka 消费、API 回调等接口须支持重试且不产生脏数据（幂等键/去重）。
- **防御式编程**：对入参与外部依赖返回值做严格校验与异常处理。

### 2. 伸缩性（Scalability）
- **无状态设计**：Collector、查询服务等应用层尽量无状态，便于 K8s HPA 自动扩缩容。
- **数据分片**：MySQL/ES/Redis/Kafka 须支持水平拆分（Kafka 分区数、ES Index Shard、Redis 分片）。
- **读写分离**：查询链路（读）与写入链路（写）分离，避免慢查询拖垮数据摄入。

### 3. 性能与延迟（Performance & Latency）
- **全链路异步化**：I/O 密集型（网络调用、磁盘读写）使用异步非阻塞模型（Netty/Reactor）。
- **多级缓存**：本地缓存（Caffeine）+ 分布式缓存（Redis），降低 DB/ES 压力。
- **计算贴近存储**：聚合尽量在 ES 内部完成，避免拉取到应用层再算。
- **空间换时间**：预计算、索引优化（ES 倒排索引、MySQL 覆盖索引）。

### 4. 可观测性（Observability）
- **三位一体**：Metrics（P95 延迟）、Logging（查错）、Tracing（定位）齐全；平台自身须 Dogfooding。
- **白盒化**：暴露内部队列长度、线程池活跃度、GC 等内部状态（黄金指标）。
- **自检能力**：提供 Health Check 接口供 LB 与 K8s 探活。

### 5. 可维护性与演进性（Maintainability & Evolvability）
- **模块化与解耦**：单一职责，严防模块间循环依赖。
- **向后兼容**：API 与 Data Schema（尤其 Kafka 消息格式）新增字段可，删改字段须版本化（v1/v2，Avro Schema Registry）。
- **自动化运维**：配置热更新、自动化部署、蓝绿/金丝雀发布。

### 6. 安全性与合规性（Security & Compliance）
- **零信任**：内外网访问均需鉴权。
- **数据脱敏**：日志与 Trace 严禁明文密码、手机号、身份证号（PII）。
- **最小权限**：服务账号仅授予必要权限（呼应决议 D CMDB 兜底）。

## 二、设计原则（SOLID 与 AKF）

- **SRP 单一职责**：Collector 只收数转发，不计算拓扑。
- **OCP 开闭**：对扩展开放、对修改关闭。采样策略新增"自适应"不应改"固定采样"代码，而应新增策略类。
- **LSP 里氏替换**：存储适配层（ES/MySQL）须遵循统一 Repository 接口。
- **ISP 接口隔离**：胖接口拆小，调用方不依赖不需方法。
- **DIP 依赖倒置**：依赖抽象，不依赖具体实现。
- **AKF 拆分**：
  - X 轴：水平复制（加机器）
  - Y 轴：功能解耦（采集/分析/存储微服务拆分）
  - Z 轴：数据分片（按 UserID 或 TraceID Hash 分库分表）

## 三、全链路追踪平台特殊原则

1. **采样即正义（Sampling is King）**：不可能存 100% Trace。明确**头部采样（Head-based）**与**尾部采样（Tail-based）**场景；错误请求（4xx/5xx）、慢请求（>2s）必采，普通请求按比例采。
2. **控制基数（Control Cardinality）**：Tag 的 Key 数量受限，Value 须为枚举或低基数；**禁止将 UserID/OrderID 作为 Tag**（放 Attribute 但不索引）。
3. **最终一致性**：拓扑图计算允许秒级/分钟级延迟，不追求强实时一致。
4. **资源消耗红线（Resource Budget）**：设计须含 Agent 资源占用红线（如 CPU < 1%、内存 < 128M），否则业务方不敢接入。

## 四、设计心法 Checklist（每个设计须自问）

1. **Scale**：流量涨 10 倍，哪里会炸？怎么扩容？
2. **Failover**：机房断了，数据会丢吗？服务能切吗？
3. **Cost**：功能多花多少机器钱？有更省办法吗？（Tracing 尤甚）
4. **Latency**：用户点击到看到结果最长多久？（目标 P95 < 1s）
5. **Ops**：出问题能一眼看到哪里坏？怎么回滚？
