> 背景：两地三中心（上海 DC1 / 苏州 DC2 + DMZ），biz/dmz 网络隔离。决议 A 明确：**所有数据汇聚至上海 DC1 Kafka 集中处理与存储**。

---

## 1. 多中心拓扑与数据流向（决议 A）

```
上海 DC1（主中心，集中存储/计算）
   ├─ biz 区 Collector 集群（接收上海业务）
   └─ 汇聚 Kafka（上海 DC1）
苏州 DC2
   ├─ biz 区 Collector 集群（接收苏州业务）
   └─ 专线 → 上海 DC1 Kafka（跨中心汇聚）
DMZ 区（各中心）
   └─ 代理转发 → 本中心 biz Collector（网络隔离，决议 A）
```

- **采集就近、存储集中**：各中心本地 Collector 负责接收与一级处理（降网络延迟、提可用性），跨中心仅传输**已采样/已处理**数据至上海 DC1，节省专线带宽（成本原则）。
- **DMZ 隔离**：dmz 区 Agent 不直接出网，经本中心 dmz 代理转发至 biz Collector（零信任、网络隔离）。

## 2. 跨中心 Trace 拼接

- **TraceID 全局唯一**：采用 DD-14 类 Snowflake 策略，内嵌 `data_center` 位，保证跨中心同一次请求 TraceID 一致、可定位来源。
- **拼接逻辑**：一次跨中心调用（上海服务 → 苏州服务），同一 `trace_id` 的 Span 由上海 DC1 汇聚后按 `trace_id` 重组完整链路（Kafka 分区按 `trace_id` hash 保序，DD-02 §4）。
- **多中心维度标注**：每条 Span 携带 `data_center/network_area/room/cluster`（DD-03.5、DD-02），跨中心边在拓扑（DD-16）中以"跨中心连线"呈现。

## 3. 带宽与传输治理（成本 + 性能）

| 策略 | 说明 |
|---|---|
| 本地预处理 | 各中心 Collector 完成采样/脱敏/转换后再跨传，仅传必要数据 |
| 压缩 | Kafka 生产者启用压缩（lz4/snappy），专线流量降 50%+ |
| 限流 | 跨中心出口限流，避免抢占业务专线（配额可配） |
| 断点续传 | 专线抖动时本地磁盘缓冲（有界），恢复后重传，不丢（可靠性） |

## 4. 容灾与高可用（Failover）

- **Collector 多实例**：各中心 Collector 集群化，单点故障自动切换（无状态 HPA，DD-07）。
- **Kafka 跨副本**：上海 DC1 Kafka RF≥3；关键 Topic 可跨可用区副本（决议 A 集中，但副本跨 AZ 防单 AZ 故障）。
- **存储容灾**：ES/MySQL 多副本 + 备份（DD-04 RPO/RTO）；上海 DC1 为主，异地（苏州）可异步容灾备份（待二期评估）。
- **故障降级**：某中心 Collector 全阻 → 该中心数据本地缓冲，恢复后补传；聚合展示侧标注"数据延迟"（最终一致性，不阻塞其他中心）。

## 5. 数据一致性与合规

- **最终一致性**：跨中心数据允许秒级/分钟级延迟（拓扑、聚合），不追求强实时（追踪特殊原则）。
- **合规**：跨中心传输 TLS 加密；PII 已在源头脱敏（DD-14），跨传不泄漏；敏感配置加密（DD-09）。

## 6. 容量与扩缩（伸缩性）

- 各中心 Collector 按本地业务量独立 HPA（X 轴）；上海 DC1 汇聚 Kafka/ES 按总流量扩容（DD-04 §5）。
- 新增中心：部署本地 Collector + 专线汇聚即可接入，无中心数上限（Y/Z 轴拆分）。

## 7. 设计原则遵循声明

- 可靠性✅（本地缓冲不丢、副本容灾、降级） 伸缩性✅（各中心独立 HPA、中心可水平加） 性能✅（就近采集降延迟、预聚合） 成本✅（集中存储降本、压缩限流省带宽） 最终一致性✅（跨中心秒级延迟） 零信任✅（专线 TLS、DMZ 代理、最小权限） 控制基数✅（多中心维度低基数）。
