## 0. 全局拓扑产品需求界定（评审决议 D2）

> 评审会依据业务方对"全局拓扑"的重新阐述，明确如下产品需求与数据落地方式，作为本专题的基线约束。

- **入口与范围**：用户登录平台后点击"**全局拓扑**"菜单栏，展示**所有项目、所有工程**的请求调用拓扑关系，呈现不同**机房 / 数据中心 / 系统 / 工程**之间的访问关系，重点展示**该公司不同系统之间的调用关系**。
- **默认视图（先系统、后工程）**：
  1. **首先展示跨机房 / 跨数据中心 / 跨系统的调用关系**——即"全局系统级拓扑"，凸显不同系统之间的跨机房、跨数据中心访问（呼应两地三中心、决议 A 集中上海 DC1）。
  2. **点击具体系统后，下钻展示该系统下的工程（engineering）与其他工程之间的调用关系**——由系统级细化到工程级。
- **数据落地（重新聚合 + 落库 ES）**：平台通过 **topology-calc 重新聚合计算**，将拓扑关系**保存到 ES**——实时边写入 `npm_online_topology`（ES，1m 桶，180d），全量图合并后落入 `t_topology_snapshot`（MySQL）+ `topology:cache:{scope}`（Redis #3，TTL 1m）；前端仅检索轻量聚合（原则"计算贴近存储"）。
- **前端渲染（DD-15 协同）**：评审决议**推荐方案 A**（G6 v5 WebGL + 视口 LOD + 按 scope 聚类收起 + Web Worker 布局）；方案 B（默认聚类收起、仅下钻展开）作为降级兜底共存。

## 1. 拓扑数据模型（跨机房 / 跨数据中心）

### 1.1 节点（Node）
来源 `service_name` / `service_instance` 及 `npm_online_topology.target_type`：

| 节点类型 | 标识 | 维度字段 |
|---|---|---|
| 应用/服务 | service_name + instance | data_center / network_area / room / cluster / env |
| 数据库 | target_type=db + target_name | db_system / 所属 room |
| 消息队列 | target_type=mq + target_name | messaging.system |
| 缓存 | target_type=cache + target_name | redis 等 |

- 每个节点携带多中心维度（`data_center/network_area/room/cluster`），支撑按中心/机房切片与跨中心连线（DD-08）。

### 1.2 边（Edge）
来源 `npm_online_topology`：

| 字段 | 说明 |
|---|---|
| source_service / source_instance | 调用方 |
| target_type / target_service / target_name | 被调方（service/db/mq/cache） |
| call_count / error_count / avg_duration / p95_duration | 调用指标（1m 快照） |
| snapshot_time | 快照时间 |
| data_center / network_area / room / cluster | 边所属多中心维度 |

- **强弱依赖**：边出呼叫占比 < **5%** 节点总出调用 → 弱依赖（虚线/淡显）；阈值可配（DD-03 §9.3）。
- **异常高亮**：边错误率 > **1%** 或 p95 > SLO → 红色高亮。

### 1.3 落库（评审决议 D2：重新聚合计算保存到 ES）
- 实时边数据：`npm_online_topology`（ES，1m 桶，180d）——由 topology-calc **按 (source, target_type, target) × 多中心维度** 重新聚合后落库（非简单转发，呼应 §0 数据落地）。
- 快照：`t_topology_snapshot`（MySQL，合并后全量图）+ Redis `topology:cache:{scope}`(#3，TTL 1m，DD-02）。
- **scope 维度覆盖**：聚合结果同时按 `scope=global/system/center/room` 物化，支撑"全局系统级默认视图 → 点击系统下钻工程级"的检索（§3/§4）。

## 2. 拓扑计算与聚合管道

```
otel-spans(Kafka)
   │  topology-calc 服务消费（SRP，DD-01 §5）
   ▼
按 (source, target_type, target) × 多中心维度 聚合 1m 边
   │  → npm_online_topology（ES）
   ▼
合并快照（topology.merge.interval，默认 60s 可配，DD-03 §9.3）
   │  → t_topology_snapshot + Redis topology:cache(#3)
   ▼
前端 /topology 查询（scope=system/center/room）
```

- **计算贴近存储**：边聚合在 ES Ingest/Transform 或 topology-calc 完成，前端仅检索轻量聚合（性能原则，P95<1s）。
- **多级缓存**：`topology:cache`(Redis #3) 缓存快照，1m TTL，降低 ES 压力。
- **最终一致性**：合并频率默认 60s，拓扑允许秒级延迟（追踪特殊原则）。

## 3. 多中心 Trace 拼接后的拓扑合并逻辑（DD-08）

- **跨中心边**：同 `trace_id` 跨中心调用，Span 携带各自 `data_center`；topology-calc 按 `source.data_center → target.data_center` 生成**跨中心边**，在拓扑图中以特殊样式（如跨中心连线 + 中心标签）呈现。
- **合并策略**：各中心本地拓扑独立计算 → 上海 DC1 汇聚（决议 A）→ 按 `scope` **重新聚合**（评审决议 D2 落库 ES）：
  - `scope=global`（**全局拓扑默认入口，D2 需求**）：跨机房/跨数据中心/跨系统的调用关系，展示不同系统之间的访问关系（系统级，忽略实例与工程细节）；
  - `scope=system`：全局应用级拓扑（忽略实例）；
  - `scope=center`：按 `data_center` 分组，展示中心间调用；
  - `scope=room`：按 `room` 细化到机房级。
  - **下钻（D2 需求）**：点击 `scope=global/system` 中某具体"系统"节点 → 下钻至该系统的**工程（engineering）级**，重新聚合展示"该系统下的工程与其他工程的调用关系"（按 `engineering_id` 维度）。
- **节点去重**：同 `service_name`（跨实例）聚合为单节点，实例数作为节点属性；跨中心同名服务按 `data_center` 区分或合并（可配）。

## 4. 拓扑聚合与展示策略

| 维度切换 | 行为 |
|---|---|
| **全局拓扑（默认入口，scope=global，D2）** | 展示跨机房/跨数据中心/跨系统的调用关系，凸显不同系统之间的访问关系 |
| **点击系统下钻（system → engineering，D2）** | 进入该系统，展示其工程与其他工程的调用关系 |
| 按系统(scope=system) | 应用级拓扑，隐藏实例细节 |
| 按中心(scope=center) | 分中心布局，凸显跨中心调用 |
| 按机房(scope=room) | 机房级细化 |
| 指标叠加 | 边/节点叠加 错误率/P95/吞吐（来自 npm_online_topology） |
| 强弱依赖 | 弱依赖虚线淡显（<5% 出调用） |
| 异常高亮 | 错误率>1% 或 p95>SLO 红显 |

- **权限过滤**：按 CMDB 数据权限（DD-09）过滤可见节点/边，无权限置灰/隐藏（DD-15 PermissionGate）。

## 5. 拓扑图渲染与交互（DD-15 协同）

| 交互 | 实现 |
|---|---|
| 缩放/平移 | 画布缩放（d3-force/G6/Cytoscape），跨中心分层坐标 |
| 下钻 | 节点→实例列表→单实例指标/关联 Trace |
| 过滤 | 按 service/center/room/异常状态过滤 |
| 时间回放 | 选择历史 snapshot_time，回放拓扑演变（t_topology_snapshot） |
| 节点详情 | 点击→健康评分(DD-03 §9.1)、依赖、TopN 接口、关联 Trace |
| 跨中心可视化 | 中心分区域布局 + 跨中心连线标注 |

- **渲染性能**：万级节点采用分层/聚合懒加载（DD-15 §4）；缓存快照降重绘。

## 6. 设计原则遵循声明

- 最终一致性✅（60s 合并、秒级延迟） 计算贴近存储✅（ES 内/聚合服务算边） 多级缓存✅（Redis 拓扑缓存） 性能✅（P95<1s、懒加载） 伸缩性✅（scope 分片、中心可水平加） 零信任✅（CMDB 权限过滤） 可维护性✅（topology-calc 独立 SRP）。
- **待评审**：
  1. 跨中心同名服务合并还是区分（按 data_center 区分默认？）；
  2. 拓扑节点规模上限（万级）下的渲染与计算性能保障——**前端渲染已决议 D2 推荐方案 A（G6 WebGL+LOD+聚类）+ B 降级兜底**；后端计算通过按 `scope` 重新聚合降维（§0/§1.3/§3），仍需 PoC 验证万节点帧率与聚合耗时；
  3. 时间回放的历史快照存储粒度（1m 快照 180d 存储成本，DD-04，D9 冷层）；
  4. 强弱依赖 5% 阈值与异常 1% 阈值是否按业务分级可调。
- **评审决议 D2（已落地）**：全局拓扑需求界定（§0）、重新聚合落库 ES（§1.3/§3）、默认全局→系统下钻工程（§3/§4）、前端渲染推荐方案 A（DD-15 §7/§14）；本专题其余待评审项按上表推进。
