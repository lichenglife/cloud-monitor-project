## 1. 决议背景与问题

DD-06 v0.2 已定义 `alert-engine` 消费预聚合指标流 + 滑动窗口的评估形态，但两项关键决策仍待定，直接影响告警时效、误告率与系统复杂度：

1. **评估引擎数据来源**：是流式消费预聚合指标流（低延迟），还是定时轮询 ES（实现简单但延迟高、压 ES）？
2. **异常检测算法**：静态阈值 / 同环比-EWMA / 机器学习，一期选哪条路线？

评审会依据"可靠性/安全红线优先、成本与复杂度次要"原则给出**推荐方案**并决议。

## 2. 决议结论（D6）

| 决策项 | 决议结果 |
|---|---|
| **评估引擎（数据来源）** | **方案 X：流式消费预聚合指标流**（一期推荐）。alert-engine 订阅 Kafka 预聚合 1m 指标流（或聚合结果），内存滑动窗口评估，不压 ES。 |
| **异常检测算法（一期）** | **方案 ①：静态阈值**（规则 + `for` 持续时间）。检出延迟最低（P0≤10s），可解释、易调参。 |
| **算法演进路线** | ① 静态阈值（一期）→ ② 同环比 / EWMA（二期，自适应基线、降误告）→ ③ 机器学习（远期，需大量标注）。 |
| **投递/对接** | alert-engine 与统一告警平台解耦：本平台负责"判定 + 落库 + 事件推送"，统一平台负责最终投递/值班/升级（Y 轴拆分）。 |

> 复审各候选方案的差异与影响见决议清单 D6 对比表；本决议文档为落地级细化。

## 3. 评估引擎：流式消费预聚合流（方案 X）详细设计

### 3.1 为什么选 X 而非定时轮询 ES

| 维度 | X：流式消费预聚合流（选定） | Y：定时轮询 ES |
|---|---|---|
| 检出延迟 | 低（随流评估，P0≤10s） | 高（受轮询周期 + ES 聚合耗时约束） |
| ES 压力 | 低（仅写，不额外查） | 高（高频聚合查询，告警多规则叠加放大） |
| 实时性 | 秒级、贴合追踪场景 | 分钟级、滞后 |
| 复杂度 | 中（需预聚合流 + 消费幂等） | 低（直接查） |
| 结论 | **选定**：告警时效是可靠性红线 | 不采用 |

### 3.2 数据流架构

```
预聚合指标流（Kafka：otel-metrics / 聚合结果）
        │  alert-engine 订阅（按规则/作用域分片，X 轴 HPA）
        ▼
  ┌─────────────────────────────────────────┐
  │ alert-engine（SRP 单一职责，无状态）        │
  │   ① 规则加载（t_alert_rule + Redis 缓存）  │
  │   ② 滑动窗口评估（窗口=规则 for 时长）       │
  │   ③ 命中 → 候选事件 → 收敛（§4）           │
  │   ④ 落库 npm_online_event + 推送 alert-events│
  └─────────────────────────────────────────┘
        │                          │
        ▼                          ▼
  npm_online_event（ES 365d）   统一告警平台（Webhook/Kafka，投递/值班/升级）
        │
        ▼
  用户（DD-15 告警页 / 大屏）
```

- **预聚合前置**：指标在采集/聚合服务侧已按 1m 桶聚合（DD-02 6 索引：`npm_online_performe` 错误率/P95/吞吐、`npm_online_metric` JVM/CPU、`npm_online_topology` 边异常），alert-engine 不重复聚合（性能原则）。
- **幂等与续算**：评估状态外置 Redis，重启可续算；同一 `rule+service+instance` 收敛窗口内仅产 1 事件。

## 4. 滑动窗口与评估模型

| 要素 | 取值 | 说明 |
|---|---|---|
| 评估周期 `evaluate.interval` | P0=10s，P1/P2/P3=1m | 高频规则优先 |
| 滑动窗口 | = 规则 `for` 时长（1m/5m 可配） | 窗口内持续命中才触发，避免抖动误告 |
| 条件类型 | 阈值（> / < / 区间）、同环比、EWMA 突增、缺失（心跳丢失） | 一期以阈值为主 |
| 收敛 `group_window` | 5m（同一 rule+service+instance 仅 1 事件，计数累加） | 防告警风暴 |
| 升级 `ack_timeout` | 10min（P0 未确认自动升级二线/值班经理） | 防止漏处理 |

- **缺失类检测**：以 `heartbeat_timeout`(60s) + `for` 判定，延迟 ≤1~2 分钟（如 Collector 心跳丢失、Agent 离线）。

---

## 5. 异常检测算法选型矩阵与演进

| 维度 | ① 静态阈值（选定·一期） | ② 同环比 / EWMA（二期） | ③ 机器学习（远期） |
|---|---|---|---|
| 检出延迟 | 低（P0≤10s） | 中 | 中 |
| 误告率 | 中（需调阈值） | 低（自适应基线） | 最低 |
| 数据积累 | 无需 | 需历史 | 需大量标注 |
| 复杂度 | 低 | 中 | 高 |
| 适用期 | 一期全覆盖 | 二期稳态业务 | 远期核心链路 |

**演进原则**：一期用静态阈值快速覆盖全部 P0/P1 告警，保"不漏"；二期对稳态核心业务引入同环比/EWMA 自适应基线，降误告；远期对核心链路试点 ML 异常检测。各期可并存，不互斥。

## 6. 检测延迟 SLA（量化）

| 规则级别 | 评估周期 | 检出延迟 | 收敛窗口 |
|---|---|---|---|
| P0 | 10s | ≤ 1 评估周期（~10s）+ 收敛（5m 内仅 1 事件） | group_window=5m |
| P1 / P2 / P3 | 1m | ≤ 1~2 分钟 | group_window=5m |
| 缺失类 | 60s | ≤ 1~2 分钟 | for 判定 |

- P0 目标：核心服务错误率>10%、Collector 全阻、跨中心链路中断等，**5min 响应 SLA**（电话+IM+大屏红闪）。

## 7. 与统一告警平台对接边界

| 责任 | 本平台（alert-engine） | 统一告警平台 |
|---|---|---|
| 判定 | ✅ 规则评估、命中、收敛 | — |
| 落库 | ✅ `npm_online_event`（可回溯） | — |
| 推送 | ✅ 经 `alert-events`(Kafka) 推事件 | 订阅消费 |
| 投递 | — | ✅ 渠道（IM/短信/电话/Webhook）、值班、升级 |
| 去重 | ✅ 本平台内按 rule+service+instance 收敛 | 平台侧做跨系统去重（如有） |

- **解耦收益**：评估与投递分离（Y 轴），告警逻辑变更不影响投递通道；Webhook 鉴权 HMAC 签名（零信任）。
- **状态闭环**：事件 `status` ∈ {firing, acknowledged, resolved}；恢复事件反向更新 `resolved_time`，避免"已恢复仍红"。

---

## 8. 可靠性与伸缩性

- **不漏不误**：评估/投递解耦、事件落库可回溯；P0 平台自监控（Dogfooding，DD-05/DD-07）优先于业务告警。
- **伸缩性**：alert-engine 无状态，按规则/作用域分片并行评估（X 轴 HPA）；评估状态外置 Redis 支持重启续算。
- **降级**：预聚合流中断时，短期回退定时拉取 ES 最近窗口（保底不漏），恢复后切回流式。
