## 1. 项目背景与业务价值

公司可观测性能力分散在多个自研平台：**CMDB**（基础配置）、**日志平台**（日志采集与查询）、**统一监控告警平台**（基础设施/中间件监控）、**容器平台**（K8S + Prometheus 资源监控）。**应用链路层（Trace）监控缺失**，导致跨系统、跨中心故障定位依赖人工串联日志与监控，平均定位耗时长、成本高。

本项目建设**全链路追踪分析平台**，以 OpenTelemetry（OTel）为统一采集标准，整合 Trace / Metric / Log 三类数据，构建系统内、跨系统、多中心的全局链路拓扑视图，实现故障快速定位与资源成本优化。

> 业务价值：缩短 MTTR、夯实系统关联分析能力、消除链路数据孤岛、为容量与成本优化提供数据底座。

## 2. 项目目标与成功标准（SMART）

| 编号 | 目标 | 成功衡量标准 |
|---|---|---|
| O1 | 一期建成链路采集与多中心集中能力 | 10 套试点系统、300 工程、上海/苏州两地三中心完成接入；峰值 QPS≈500 稳定采集 |
| O2 | 构建全局拓扑与全链路分析 | 支持系统内/跨系统/多中心拓扑、Trace-Metric-Log 三域关联、瀑布图/代码栈/日志关联 |
| O3 | 端到端查询性能达标 | Trace 查询延迟 P95 < 1s；全量数据存储周期 1 年 |
| O4 | 实现 Agent 可控治理 | 管理端热配置采样率/最大 TPS，≤30s 下发生效，无需重启 Agent |
| O5 | 规范过程与质量基线 | 阶段评审门 100% 通过；缺陷逃逸率 ≤ 目标阈值；CMMI5 关键过程域落地 |

## 3. 项目范围

**一期（9 个月）纳入：**
- OTel 定制 Collector + Java(SkyWalking Agent) / Go(OTel Agent) 接入
- 两地三中心采集与上海机房1（DC1）biz 区数据集中（Kafka 管道）
- 存储分层：MySQL（元数据）+ Redis（缓存/实时/配置）+ ES（Trace 检索）+ Kafka（管道）
- 全链路分析（L1）+ L2 能力（火焰图、热点方法分析、服务健康评分）按评审决议 B 纳入一期
- 日志平台 SDK 改造复用 Agent TraceID（MDC 注入，initContainer 注入）
- CMDB 权限基线（最小权限兜底）
- 管理平台：用户/项目/集群/Agent/字典/审计管理

**二期（10 个月，不在本期范围）：** 深度合规模则与脱敏治理、更广系统接入、智能根因分析等。

**明确排除（一期）：** 深度数据脱敏合规设计、全量 200 套系统接入。

## 7. 预算与资源

- **人力**：15 人团队（PM/产品/开发/测试/运维），预算 = Σ(各工种工时 × 单价) + 服务器费用。
- **服务器**：上海 DC1 biz 区集中部署（含 Redis/MySQL/Kafka/ES/平台应用）。
- **预算申报**：按整体项目工时 × 工种工资 + 服务器费用。

## 8. 项目约束与假设

- **约束**：两地三中心、同机房 biz 与 dmz 网络不通（dmz 经代理转发）；数据集中上海 DC1（决议 A）；存储周期 1 年、P95<1s。
- **假设**：CMDB 具备拓扑构建基础数据（待二次评估）；试点系统配合改造窗口充足。
