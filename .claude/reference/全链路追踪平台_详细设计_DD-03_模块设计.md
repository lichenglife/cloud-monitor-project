## 2. 模块一：数据采集（3.1）

- **功能**：Collector 接收 OTLP（gRPC:4317 / HTTP:4318）→ 解析 → 采样（头部/尾部）→ 转换 → 导出 Kafka；Agent（Java=SkyWalking / Go=OTel）注入与启动参数；多中心管道（biz 直连 / dmz 代理转发，决议 A）；数据集中上海 DC1（决议 A）。
- **处理流程**：
  1. Agent 注入（K8s initContainer / VM 手动）→ 上报 OTLP 至 Collector（biz 直连或 dmz 代理）。
  2. Collector 入口背压 + 幂等去重（`trace_id+span_id`，Redis #4）→ 采样决策（错误/慢请求必采，普通按比例；自适应见 DD-14）。
  3. 导出 `otel-spans/metrics/logs` 至 Kafka（上海 DC1）。
  4. 消费侧（topology-calc / 聚合服务）写入 ES 各索引。
- **接口依赖（DD-01）**：`POST /agents/config`（热配置≤30s）、`POST /agents/upgrade`、`GET /agents/compatibility`、`GET /collectors`、`GET /ingest/*`、`GET /ingest/health`。
- **数据依赖（DD-02）**：Redis `agent:config:{clusterId}`(#1)、`trace:dedup:{traceId}`(#4)；Kafka `otel-spans/metrics/logs`；ES `trace_npm_jaeger`；MySQL `t_agent_instance`、`t_agent_config`、`t_agent_registration`、`t_engineering`。
- **安全**：Agent 注册令牌（`t_agent_registration.token`），传输 TLS；Collector 仅接受鉴权 Agent；最小权限（CMDB 兜底）。
- **设计原则遵循声明**：可靠性✅（背压+降级+幂等） 无状态✅（Collector 可 HPA） 采样即正义✅ 资源消耗红线✅（Agent CPU<1%/内存<128M，见 DD-13） 防御式编程✅ 零信任✅。
- **待评审**：Collector 内部管线线程模型（receive/process/export 线程池配比，见 DD-07 §8）。尾部采样缓冲时长已决议 **D5：window=10s / in-flight=20000**（见 DD-07 §3.2），不再待评审。

## 3. 模块二：监控与可视化（3.2）

- **功能**：概览大屏（全局健康/核心指标）、拓扑计算（强/弱依赖识别、跨中心合并）、Dashboard 配置。
- **处理流程**：`topology-calc` 消费 `otel-spans` → 聚合为 `npm_online_topology`（边 `target_type`∈{service,db,mq,cache}）→ 多中心合并快照 → 写入 `t_topology_snapshot` + Redis `topology:cache:{scope}`；概览聚合 `npm_online_metric`/`npm_online_performe`。
- **接口依赖**：`GET /dashboard/overview`、`GET /topology`（scope=system/center/room）、`GET /services`、`GET /services/{name}`。
- **数据依赖**：ES `npm_online_topology`/`npm_online_metric`/`npm_online_performe`；Redis `topology:cache`(#3)；MySQL `t_topology_snapshot`。
- **UI**：拓扑图（DD-16，跨中心布局/缩放/下钻/过滤/时间回放）；概览大屏卡片。
- **安全**：拓扑按 CMDB 数据权限过滤可见范围。
- **设计原则遵循声明**：最终一致性✅（拓扑秒级延迟） 计算贴近存储✅（ES 内聚合） 多级缓存✅（Redis 拓扑缓存） 性能✅（P95<1s）。
- **已决议（见 9.3）**：拓扑快照合并频率默认 60s（可配），强弱依赖/异常阈值已定。

## 4. 模块三：全链路分析（3.3）

- **功能**：Trace 列表查询/筛选、Trace 详情（瀑布图/代码栈）、Trace-Metric-Log 三域关联。
- **处理流程**：
  1. 查询 `trace_npm_jaeger`（按 service/status/time/duration 过滤）→ 组装瀑布图与代码栈。
  2. 三域关联：同 service+时间窗关联 `npm_online_metric`；按 `traceId` 关联外部日志平台（`/logs?traceId`）。
- **接口依赖**：`GET /traces`、`GET /traces/{traceId}`、`GET /traces/{traceId}/metrics`、`GET /traces/{traceId}/logs`、`GET /traces/error-analysis`。
- **数据依赖**：ES `trace_npm_jaeger`、`npm_online_metric`；外部日志平台（按 TraceID）。
- **UI**：Trace 列表、瀑布图（Span 时序/耗时条）、代码栈树、三域关联面板。
- **安全**：Trace 属性 PII 脱敏（DD-14、DD-03.5）。
- **设计原则遵循声明**：性能✅（P95<1s，trace 索引优化） 控制基数✅ 防御式编程✅。
- **已决议（见 9.4）**：超长 Trace 截断（采集 5 万/展示 1 万）、三域关联时间窗默认 5min。

## 5. 模块四：应用性能 APM（3.4）

- **功能**：火焰图、热点方法分析、服务健康评分、接口性能 TopN。
- **处理流程**：
  - 火焰图：`trace_npm_jaeger` 调用栈按 `parent_span_id` 聚合为层级耗时（计算贴近存储，或预聚合）。
  - 热点方法：按 `operation_name`/代码栈属性聚合耗时与调用次数（读 `npm_online_performe`）。
  - 健康评分：多指标加权（错误率/时延 P95/饱和度，来自 `npm_online_metric`/`npm_online_performe`）→ 0~100 分，写入 Redis `health:score`(#9)。
  - 接口 TopN：`npm_online_performe` 按 endpoint 排序。
- **接口依赖**：`GET /services/{name}/health`、`/flamegraph`、`/hotspots`、`/top-apis`、`/apm`。
- **数据依赖**：ES `npm_online_metric`（评分输入）、`trace_npm_jaeger`（火焰图源）、`npm_online_performe`（TopN）；Redis `health:score`(#9)。
- **UI**：火焰图（DD-15 可视化）、热点列表、健康评分仪表、TopN 表格。
- **安全**：同模块三脱敏。
- **设计原则遵循声明**：计算贴近存储✅ 性能✅ 多级缓存✅（健康评分缓存）。
- **已决议（见 9.1/9.2）**：健康评分权重模型 v1（错误35%/时延25%/资源20%/告警20%）+ 二期拖拉拽配置；火焰图/热点聚合粒度与采样 v1。

## 6. 模块五：业务与基础设施监控（3.5）

- **功能**：指标接入、与统一监控告警平台/容器平台对接（基建/容器/中间件指标）。
- **处理流程**：统一监控平台、容器平台提供基础设施/容器指标 → 经专家评审（§9.5）已决议**落库扩展为 `npm_online_infra`**（ES 索引 + Kafka `otel-infra-metrics` + Redis #13/#14 + MySQL `t_infra_resource`，结构见 DD-02 v1.1）。**实施范围按排期**：一期可先对接查询（控成本快速上线），二期落库提升查询效率；或资源充足时一期即落库，以 DD-02 v1.1 结构为准。
- **接口依赖**：`GET /infra/metrics`、`GET /container/metrics`、`GET /middleware/metrics`（DD-01，已就绪）。
- **数据依赖**：外部统一监控/容器平台（对接查询）；落库后读 ES `npm_online_infra`（DD-02 v1.1）。
- **UI**：基建/容器/中间件监控页（指标卡片 + 趋势）。
- **安全**：对接接口鉴权、数据权限。
- **设计原则遵循声明**：成本✅（一期可先对接控成本、二期落库） 解耦✅（Y 轴微服务拆分） 零信任✅。
- **已决议（见 9.5，与 DD-02 v1.1 一致）**：扩展 `npm_online_infra` 落库，实施范围待排期；本段已对齐，消除原"不独立落库"表述与落库决议的矛盾。

## 8. 跨模块依赖与设计原则总览

- **依赖主线**：`数据采集`(写) → Kafka → 聚合服务(写 ES 6 索引) → `监控/分析/APM`(读) + `日志`(外部关联)。
- **原则总览**：可靠性（背压/降级/幂等）、伸缩性（无状态/分片）、性能（P95<1s/预聚合/多级缓存）、可观测性（平台自身黄金指标 DD-05）、可维护性（模块解耦/SRP）、安全（零信任/脱敏/最小权限）均在各模块落地；追踪特殊原则（采样即正义/基数控制/资源红线/最终一致性）贯穿。

## 9. 专家评审结论与决议（原待评审项）

### 9.1 健康评分权重模型（模块四）
**一版权重模型（v1）**：综合四维，每维 0~100 惩罚分，按权重折算后从 100 扣除。

| 维度 | 权重 | 惩罚分计算（0~100） | 数据来源 |
|---|---|---|---|
| 错误率 | 35% | `min(100, errorReq_5min/totalReq_5min × 200)` | `npm_online_performe`(error/call) |
| 时延劣化 | 25% | `clamp((p95 − slo_p95)/slo_p95, 0,1) × 100` | `npm_online_performe`(p95) |
| 资源使用率 | 20% | `max(cpu_util, mem_util, gc_pressure)` 归一（>85% 趋近 100） | `npm_online_metric`(JVM/CPU) |
| 告警事件 | 20% | `min(100, Σ sev_w×cnt)`，P0=40/P1=20/P2=10/P3=5（近5min） | `npm_online_event` |

`health = clamp(100 − (E×0.35 + L×0.25 + R×0.20 + A×0.20), 0, 100)`
分级：90~100 健康 / 70~89 关注 / 40~69 警告 / <40 严重。结果缓存 Redis `health:score`(#9)，1m TTL。
**说明**：权重为经验初版，聚焦用户可感知的错误与慢请求（合计 60%）。
**二期能力**：该模块支持用户**拖拉拽画图配置**权重模型——DD-15 可视化编排 + 新增 `t_health_model` 配置表，用户自定义维度/权重/阈值，模型以 JSON 存储、热加载，无需改码。

### 9.2 火焰图/热点聚合粒度与采样（模块四）
参考大厂（Jaeger / Pyroscope / EagleEye）给出一版：
- **火焰图**：以 `trace_npm_jaeger` 为源，按 `parent_span_id` 在 (service, 1m bucket) 内构建调用层级；**不呈现每条 Trace**，而是选取**代表 Trace**（桶内按时延中位数或 Top-K 慢 Trace，K 默认 100）聚合为火焰图，避免全量渲染。
- **热点方法**：按 `operation_name`/代码级 span 属性聚合 (service, 1m/5m)，排序指标为 **self_time**（duration − Σchildren，自身耗时）与调用次数，输出 TopN。
- **采样**：火焰图采样 Top-K 慢 Trace（K=100 默认可配）；热点基于 1m 预聚合（与 `npm_online_performe` 同源），高基数降采样。
- **输出缓存**：火焰图层级 JSON 缓存 Redis（或落 `npm_online_hotspot` 索引），保障 P95<1s。
- **迭代**：粒度（方法级/类级）与采样率按实际场景后续调优。

### 9.3 拓扑快照合并频率（模块二）
- **可配置**：参数 `topology.merge.interval`（秒），经 config-center 热配置（Redis #1）下发，默认 **60s（1m）**。
- **强弱依赖判定**：边出呼叫占比 < **5%** 节点总出调用 → 弱依赖（虚线/淡显）；错误率 > **1%** 或 p95 > SLO → 异常高亮（红）；阈值可配。
- 快照写入 `t_topology_snapshot` + Redis `topology:cache`(#3)，最终一致性（秒级）。

### 9.4 超长 Trace 截断与三域关联时间窗（模块三）
- **超长 Trace 截断**：采集侧硬上限 **50,000 spans/Trace**（Collector 超限丢弃并计数）；展示侧软上限 **10,000 spans**，超出保留 错误/慢 span + 采样子集，标记 `truncated=true`。
- **三域关联时间窗**：Metric 关联默认窗口 **[start−5min, end+1min]**（环绕 Trace 时间）；Log 关联按 **精确 traceId** 匹配（无需时间窗）；窗口值可配（默认 5min）。

### 9.5 基础设施数据扩展落库（模块五）
为满足更高效率查询，将基建/容器/中间件指标**落库 + 预聚合 + 进 Redis**，重新定义结构：
- **ES 新增 `npm_online_infra`**（或按资源类型拆 `…_host/_container/_middleware`）：文档 `(resource_type, resource_id, metric_name, time_bucket=1m) → value/unit`，维度 `cluster_name/room/data_center/network_area/env/host/pod/instance`。
- **Kafka 新增 `otel-infra-metrics`**：统一监控/容器平台(Prometheus) → 采集适配层 → 该 topic → 聚合服务写 ES。
- **Redis 新增 `infra:rt:{resourceId}`(ZSet 实时) / `infra:cache:{scope}`(看板)**，复用多级缓存。
- **MySQL 新增 `t_infra_resource`**（资源注册：host/pod/instance 元数据，关联 room/cluster，支撑权限与拓扑）。
- **DD-01 接口已就绪**：`/infra/metrics`、`/container/metrics`、`/middleware/metrics` 直接服务前后端页面。
- 落库后查询走预聚合索引（P95<1s），不再实时拉取外部，效率显著提升。
- **同步更新 DD-02**：新增上述索引/表/Kafka topic/Redis key（DD-02 升至 v1.1）。
- 实施范围（一期/二期）由排期确认；设计本版冻结。
