## 2. 采样率动态调整（采样即正义 + 成本）

### 2.1 静态采样（基线，呼应 DD-07）
- 由 `t_agent_config.sample_rate` 配置，头部按比例（默认试点 100%，后续按成本下调）。
- 尾部强制：错误/慢(>2s)必采（DD-07 §3.2）。

### 2.2 自适应采样（Adaptive Sampling，进阶）
- **触发维度**：
  - 流量维度：入口 QPS 超阈值 → 自动下调 `sample_rate`（如 100%→20%）控成本；
  - 错误维度：错误率升高 → 自动上调采样率保错误链路（排查需要）；
  - 耗时维度：P95/P99 百分位异常 → 上调慢请求采样。
- **实现**：alert-engine / config-center 基于 `npm_online_performe` 派生指标计算目标 `sample_rate`，经 config-center 下发 `agent:config`(#1)，Agent ≤30s 生效（DD-07 §6）。
- **约束**：下调有下限（保错误/慢必采 + 最小可观测样本），避免"省到看不见"；调档平滑（避免抖动，如每次 ±10% 步进）。
- **评审决议 D4=方案A（一期）**：自适应采样采用**规则触发调档**——错误率↑→采样↑、流量↑→采样↓、耗时百分位异常→上调慢请求采样，步进 ±10%；**方案 B（闭环 PID 连续调节）作二期演进，方案 C（静态）作兜底**。触发阈值在 §7 待评审项收敛。

### 2.3 成本护栏（成本原则）
- 单工程月存储/算力预算（可配），超预算自动降采样并告警（DD-06）。
- 聚合索引降采样（1m→5m→1h，DD-02 §3.5 ILM），原始 Trace 超 1 年删除（DD-04）。

## 3. TraceID 生成策略（唯一 / 防冲突 / 易排查）

> 类 Snowflake 思想，兼顾全局唯一、跨中心拼接、时间有序、便于定位。

```
TraceID (128bit) 结构（D7 决议，2026-07-20 补正为真 128bit）：
┌──────────── 高 64bit（新增·续位+随机）────────────┐ ┌──────────── 低 64bit（原 D7 字段·零改动）────────────┐
│ 时间戳续位(23bit,当前恒0) │ 随机数(41bit,CSPRNG) │ │ 时间戳毫秒低(41bit) │ 数据中心(3bit) │ 节点(10bit) │ 序列(10bit) │
└──────────────────────────┴──────────────────────┘ └─────────────────────┴────────────────┴─────────────┴─────────────┘
```
- **时间戳**：毫秒级 `epoch_ms`，低 41bit 存储（~69 年），高 23bit 续位预留当前恒 0；保证时间有序（ES `start_time` 排序、按时间排查）。
- **随机 41bit**：CSPRNG 填充，单中心同毫秒/同节点/同序列号极端碰撞下的全局唯一兜底（2^41 取值空间）。
- **数据中心/节点位**：内嵌 `data_center`/`room` 信息，跨中心 Trace 拼接时天然可定位来源（呼应 DD-08）；**仅作来源标识，非实例主键**。
- **序列号**：同毫秒内自增，防同刻碰撞。
- **生成点**：Agent 入口生成（Java=SkyWalking 自定义 TraceID generator；Go=OTel SDK `IDGenerator`），Collector 不重生成仅透传/校验；经 Kafka 分区 `trace_id` hash 保序（DD-02 §4）。
- **节点位容量评估（后端架构评审）**：v1 草案 `实例/节点(10bit)` = 每中心 **1024 个节点**。面对 5000+ 工程、单中心可能数万 Agent 实例，**10bit 不足以唯一标识实例**——但 TraceID 节点位仅用于"定位来源/排查"，**不要求全局唯一**（同毫秒序列号 10bit 已防同刻碰撞；实例维度靠 `service_instance`/`host_ip` 字段精确标识）。若需节点位更强区分，二期可扩至 12bit（4096/中心）或将该位改为"Collector 集群 ID"。**结论**：v1 10bit 可接受（节点位=来源标识，非实例主键），不阻塞落地。**评审决议 D7=方案A：采用 128bit（时间戳41 + 数据中心3 + 节点10bit=1024/中心 + 序列10bit），节点位仅作来源标识，实例精确识别靠 `service_instance`/`host_ip`；方案 B（12bit）作二期、方案 C（64bit）不推荐。**
- **向后兼容**：旧 64bit TraceID 进来时高位补随机并标记 `legacy_trace_id=true`，不阻断消费（DD-03.5 §4）。

> ✅ 2026-07-20 决议补正（已闭合）：原草案标"128bit"但 4 字段和仅 64bit、高 64bit 未定义。经决议人确认，方案 A 明确为**真 128bit = 高 64bit（时间戳续位 23bit + 随机 41bit）‖ 低 64bit（原字段零改动）**，随机 41bit 兜底防碰撞并原生兼容 OTel 128bit。同步更新决议清单 D7（详情段 + 决议记录表行）。本项编码阻塞解除。

## 4. 数据清洗与 ETL（脱敏 + 降噪）

### 4.1 无效 Span 过滤（降噪降本）
- 过滤：健康检查探测 Span（`/health`、`/actuator`）、内部心跳、超长截断尾段（DD-03 §9.4 超限 5 万丢弃）。
- 合并：同一调用栈重复 Span 压缩（如批量 SQL 合并为 `db.batch` 一条 + count）。

### 4.2 敏感字段脱敏（合规红线，呼应 DD-03.5 §3、DD-09）
- **脱敏对象**：`db.statement`（参数）、`http.url`（query 参数）、`http.client_ip`、自定义 Attribute 中命中 PII 正则的内容。
- **脱敏规则（v1）**：
  | 类型 | 正则示例 | 掩码 |
  |---|---|---|
  | 手机号 | `1[3-9]\d{9}` | `138****1234` |
  | 身份证 | `\d{17}[\dXx]` | 前 6 后 4 保留，中间 `*` |
  | 密码/token | `password|token|secret=...` | 整体 `***` |
  | 银行卡 | `\d{16,19}` | 前 6 后 4 保留 |
- **执行位置**：采集层（Agent/Collector Transform，DD-07 §4.5）**源头脱敏**，避免敏感数据落盘/外传；脱敏规则集中配置、版本管理。
- **不索引高基数**：脱敏后原文仅可能采样留存于 `attributes`(flattened)，不建索引（控制基数）。

### 4.3 降基数（控制基数原则）
- 禁止 UserID/OrderID 作为 Tag/索引字段（DD-03.5 §3）；如需业务关联，写入 Attribute 不索引。
- `statement_hash` 替代 SQL 原文（DD-02 `npm_online_db`），降基数且保留"哪类语句慢"。

## 5. Schema 治理（向后兼容）

- **Avro + Schema Registry**（DD-02 §4）：Span/Metric 消息 v1 字段全集本专题**注册锁定**；新增字段兼容（旧消费者忽略），删改字段升 `v2`。
- **字段全集（v1 核心，节选）**：
  - Span：trace_id / span_id / parent_span_id / service_name / operation_name / span_kind / status_code / start_time / duration / attributes(flattened) + 多中心维度；
  - Metric：resource / metric_name / value / timestamp + 多中心维度。
- **兼容性测试**：Schema 变更入 CI 卡点，校验上下游（Collector/聚合/ES）兼容。

## 6. 数据质量与血缘

- **质量监控**：脱敏覆盖率、采样率实际值 vs 配置、无效 Span 占比、PII 泄漏扫描（DD-05/DD-06 指标）。
- **血缘**：Trace → Kafka Topic → ES 索引 → 页面，可经 `trace_id` 全链路回溯（DD-03 §4/§5）。
