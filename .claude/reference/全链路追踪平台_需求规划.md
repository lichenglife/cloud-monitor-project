## 已确认关键决策（用户拍板）

- **建设路线**：自研整合 —— 以 **OpenTelemetry (OTel)** 为统一采集标准，整合现有自研平台，自建存储与分析层。
- **一期范围**：**直接多中心** —— 一期即覆盖多中心链路汇聚与一致性校验（M7 提级为一期 Must）。
- **现状基线**：当前**无应用链路层监控、无 OTel 采集**，仅有基础设施/容器/日志三类监控。本平台为"新建链路采集 + 整合既有指标/日志"的绿地建设。
- **合规范围**：一阶段不做深度合规/脱敏功能设计，留阶段二详细功能设计；本阶段仅做基础权限对接（CMDB 权限信息）。

## 一、项目背景

### 1.1 行业背景与对标（为什么必须做、做成什么样）
可观测性已成为大规模微服务运维的**核心基础设施**，业界共识其由**指标（Metrics）、日志（Logs）、追踪（Traces）、拓扑（Topology）"四大支柱"**构成。行业演进与标杆实践：

- **理论奠基**：Google **Dapper**（2010）确立分布式追踪"调用链 + 全局 TraceID"方法论，是几乎所有大厂链路系统的源头。
- **阿里鹰眼（EagleEye）**：还原完整调用堆栈、聚合分析，支撑故障定位、容量预估、强弱依赖识别。
- **字节跳动全栈可观测体系**：基于亿级 QPS 锤炼，融合 **OpenTelemetry + Prometheus**，推行**埋点标准化**、**Trace-Metric-Log 关联**、**持续性能剖析（Profiling）**与**全链路压测/流量染色**。
- **美团 CAT / MTrace**：秒级处理、全量采集、多语言支持，集成链路追踪 + 告警推送，保障超大规模分布式系统稳定。

> **成熟平台的共性能力（本项目对标基准）**：
> 1. **统一采集与埋点标准化**（OTel 成为事实标准）
> 2. **Trace-Metric-Log 三域关联**（一次排障不必跨系统跳转）
> 3. **实时/自动拓扑 + 强弱依赖识别**
> 4. **智能告警 + 根因分析（RCA / AIOps）**
> 5. **持续性能剖析（Profiling，火焰图定位慢方法）**
> 6. **SLO / 错误预算与稳定性体系**
> 7. **全链路压测与混沌演练（主动验证）**

**行业趋势**：监控正从"被动告警"走向"主动洞察"，从"三套系统各自为战"走向"OTel 统一采集 + 后端关联分析"。金融行业普遍以业务连续性为目标，强调端到端交易链路透视与分钟级定位（某银行案例：支付延迟 5 分钟定位至数据库锁瓶颈，运维效率提升 60%）。

### 1.2 公司现状诊断
| 现状问题 | 说明 |
|---|---|
| **缺链路层监控（核心缺口）** | 当前仅有基础设施监控、容器层监控、日志，**无应用链路层（Trace）监控**，调用链完全不可见 |
| 数据孤岛 | 指标、日志、链路分属不同平台，缺乏统一关联主键 |
| 拓扑缺失/不准 | 缺乏系统内/跨系统/多中心统一拓扑，依赖关系靠人工梳理 |
| 排障低效 | 故障定位跨多平台跳转，上下游影响分析依赖经验，MTTR 偏高 |
| 资源视角割裂 | 链路与资源指标不关联，无法做动态水位标定与成本优化 |

**现有自研平台能力盘点（对接对象）**

| 平台 | 已有能力 | 对外接口 | 对全链路平台的价值 |
|---|---|---|---|
| CMDB | 工程名、系统名、应用/运维负责人、权限、机房、数据中心 | 元数据查询 | 拓扑元数据补全、权限基线（**非调用关系来源**） |
| 日志平台 | 各系统已通过日志 SDK 对接；提供日志查询接口 | 日志查询 API | 链路 × 日志关联检索 |
| 统一监控告警平台 | 物理机/虚拟机基础设施层 + 中间件/数据库监控 | 指标/告警接口 | 链路 × 指标关联诊断、RCA |
| 容器平台 | K8S 容器化部署；Prometheus/Grafana 容器层资源监控 | 容器指标/部署信息 | 容器资源 × 链路关联、动态水位 |

> **结论**：链路层是空白，指标与日志已有底座。平台核心是**补齐链路采集 + 打通三域关联**，而非重建指标/日志。

### 1.3 关键架构约束（务必前置设计）
- **两地三中心 + DMZ**：上海、苏州两地，三个数据中心；各机房之间网络互通。
- **同机房区域隔离**：同一机房内 **biz 区域与 dm 区域网络不通** —— 直接影响 Collector/后端部署位置与跨区链路汇聚路径。
- **混合部署**：K8S（1.21/1.25）容器化为主 + 部分虚拟机；Agent 需同时支持容器注入与虚拟机进程级两种形态；中间件含 MySQL/MongoDB/Redis/Kafka/MQ/Tomcat/宝蓝德(BES)、nginx 等代理。

### 1.4 本项目定位
> 以**自研整合 + OpenTelemetry 统一采集**为路线，在现有指标/日志底座之上**补齐应用链路层**，构建"系统内 + 跨系统 + 多中心"的全局链路拓扑与三域关联诊断能力，对标大厂全栈可观测体系，分阶段达成分钟级故障定位与资源利用率可量化提升。

## 五、核心痛点与场景

### 5.1 核心痛点
- **P1 链路空白**：无任何应用链路层监控，调用链完全不可见。
- **P2 数据孤岛**：指标、日志、链路分属不同平台，缺乏统一关联主键。
- **P3 拓扑缺失/不准**：缺乏统一拓扑视图，依赖关系靠人工梳理。
- **P4 排障低效**：跨平台跳转，上下游影响分析依赖经验。
- **P5 资源视角割裂**：链路与资源指标不关联，无法做水位标定与成本优化。

### 5.2 典型使用场景
1. **故障应急**：告警触发 → 链路视图 → 定位慢调用/错误码 → 沿拓扑看上下游 → 联合指标/日志确认根因。
2. **变更影响评估**：基于拓扑识别上下游依赖，评估爆炸半径（强弱依赖区分）。
3. **容量与成本治理**：链路调用量 + 容器/资源指标关联，标定水位，识别冗余。
4. **多中心一致性核查**：跨中心链路拼接，校验多活/灾备链路正确性（应对 biz/dm 隔离）。
5. **（后续）性能剖析**：持续 Profiling 定位 CPU/内存热点与慢方法。
6. **（后续）稳定性演练**：全链路压测 / 混沌演练验证系统韧性。

## 七、优先级建议（MoSCoW）

- **Must（一期必做 · L1 全量）**：M1 统一采集（OTel+Agent 管理）、M2 统一存储（1年/<1s）、M3 链路检索、M4 全局拓扑（含强弱依赖）、M7 多中心汇聚、M15 基础权限。
- **Should（一期强化 · L1→L2 过渡）**：M5 影响分析、M5.1 告警中心、M6 三域关联、M14 统一视图。
- **Could（后续演进 · L2/L3）**：M8 成本优化、M9 持续剖析、M10 根因分析、M11 SLO、M12 全链路压测、M13 混沌演练。
- **Won't（本阶段不做）**：① 深度合规/脱敏（留阶段二）；② 替换所有现有自研平台（整合打通为主）；③ 全量 200 套接入（一期仅 10 套试点）。

## 八、实现复杂度与关键风险

| 风险点 | 说明 | 缓解建议 |
|---|---|---|
| 链路采集新建 | 现状无 OTel/链路，需从零植入探针 | 优先 OTel 低侵入方案；Java 自动埋点 + Go 手动埋点；灰度接入 |
| **同机房 biz/dm 网络不通** | agent 上报与跨区查询通道可能受阻 | **架构设计前置**明确 Collector 部署位置、跨区汇聚与查询通道（关键评审议题） |
| 多中心数据汇聚 | 跨中心网络、一致性、时钟同步 | 统一 TraceID + 中心标识；先评估网络与合规，避免数据集中落地 |
| Agent 管理复杂度 | 配置下发到海量实例的版本/配置一致性 | 配置中心 + 灰度下发 + 心跳自愈；一期仅 300 工程验证 |
| 采集性能开销 | 全量 Trace 影响业务 | 分级采样、agent 层可控采样率、异步上报 |
| 存储成本 | Trace 数据量大、留存 1 年 | 冷热分层、降采样 |
| 关联分析准确性 | 链路-指标-日志关联模型门槛 | 先规则后模型，小范围验证 |
| 现有平台对接 | CMDB/日志/监控/容器接口标准不一 | 优先标准化接口，存量做适配层 |
| 规模演进 | 一期 300 工程试点，终态 5000+ 工程 | 架构按终态容量设计，一期仅试点验证 |
